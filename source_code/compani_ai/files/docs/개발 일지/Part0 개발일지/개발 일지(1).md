## 1. 작업 내용
이번 Stage에서는 문서 중심 저장소를 실제 코드 저장소 구조로 전환하기 위해 `src` 레이아웃을 기준으로 5개 레이어 패키지를 생성했습니다.  
`domain/ports.py`에 LLM/워크스페이스/메시지큐/스토리지/이벤트버스 Protocol을 선 정의해 에이전트 구현체 간 직접 의존을 차단했습니다.  
관측 기반으로 `run_id/task_id/message_id` 생성 유틸, ErrorCode Enum, JSON 구조 로거를 구현했고, 테스트 선행 원칙에 따라 Mock 3종과 단위 테스트를 함께 작성했습니다.

## 2. 결과물
생성된 파일: `src/domain/ports.py` — Part 1~7이 공통 사용할 인터페이스 계약 정의.  
생성된 파일: `src/observability/ids.py`, `src/observability/error_codes.py`, `src/observability/logger.py` — 식별자/에러코드/구조화 로그 기반 제공.  
생성된 파일: `src/adapters/mock_llm_provider.py`, `src/adapters/mock_workspace.py`, `src/adapters/mock_message_queue.py`, `tests/unit/test_stage1_smoke.py` — Mock 기반 최소 검증 세트.  
생성된 파일: `requirements.txt`, `requirements-dev.txt`, `.env.example`, `pyproject.toml` — 의존성/환경/Python 3.11 고정값 명시.  
수정된 파일: `files/develop-rule/00_COMMON_RULES.md` — 1.1 디렉토리 구조를 `src/domain` 형태로 업데이트.

## 3. 결정 이유
코드 루트는 `Company Ai/` 루트 아래 `src/`를 두는 형태로 확정하여 기존 `files/` 문서 자산과 코드 자산의 경계를 명확히 분리했습니다.  
로깅은 `structlog` 우선 설계를 유지하되, 현재 오프라인 패키지 인덱스 환경에서도 Stage 검증이 가능하도록 표준 JSON fallback을 추가했습니다.  
전략 문서(개발 일지(4).md) Part 0 Stage 1의 산출물 목록을 그대로 따랐으며, 전략 변경 이력은 다음과 같습니다: `structlog` 미설치 환경 대응 fallback 추가(실행 환경 제약 대응 목적, 인터페이스는 유지).
