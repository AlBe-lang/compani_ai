핵심은 Part(파이프라인) → Stage(단계) → Domain(분야) 3계층으로 운영하는 것입니다.
지금 문서 상태 기준으로는 먼저 정합성부터 맞추고, 그 다음 파트를 순차 실행하는 방식이 가장 안전합니다.

1. 시작 전 정합성 정리(필수)

단일 기준 문서(SSOT)를 1개 정합니다.
문서 충돌을 먼저 해결합니다.
기준 예시:
문서 수 7개/11개 충돌: README.md (line 9), SUMMARY.md (line 12)
메모리 가정 충돌 ~16GB/~17GB: multi_agent_system_design.md (line 290), SYSTEM_ARCHITECTURE.md (line 770)
압축/파일명 안내 불일치: README.md (line 277), SUMMARY.md (line 311)

2. 작업 단위 표준 템플릿(모든 단계 공통)

입력: 선행 조건, 의존성.
출력: 코드/문서/스크립트 산출물.
검증: 테스트 명령, 성공 기준 수치.
관측: 로그 키, 메트릭, 에러코드.
완료 기준(DoD): 다음 단계로 넘어가는 조건.
이 템플릿을 쓰면 “어디서 문제났는지”를 단계 단위로 바로 추적할 수 있습니다.

3. Part별 실행 전략
DEVELOPMENT_PIPELINE.md의 3파트 구조( (line 37), (line 718), (line 956))를 그대로 사용하세요.

Part 1 개발 파이프라인
Stage 1: 리포지토리 골격, 코드 스타일, pre-commit.
Stage 2: 로컬 실행 스크립트(setup/test/lint).
Stage 3: 테스트 체계(단위/통합/E2E 분리).
Stage 4: CI 파이프라인.
Stage 5: 배포 스크립트/릴리스 규칙.
게이트: “PR에서 자동 품질 체크 100% 통과” 전에는 Part 2 금지.
Part 2 시스템 파이프라인
Stage 1: 도메인 계약 정의(WorkItem/Message/Strategy/TaskResult 스키마 고정).
Stage 2: LLM Provider Adapter(재시도/타임아웃/오류코드).
Stage 3: CTO Orchestration(전략 수립, 작업 분해).
Stage 4: SLM Execution(역할별 실행, 의존성 대기).
Stage 5: Collaboration Layer(Workspace/Event/Communication Hub).
Stage 6: End-to-End 시나리오(예: Todo 앱) 자동 검증.
게이트: 단계별 실패 지점이 로그에서 trace_id + stage_id + task_id로 바로 식별돼야 함.
Part 3 모니터링 파이프라인
Stage 1: 구조화 로그(JSON) 표준화.
Stage 2: 핵심 메트릭(성공률, 지연, 재시도, 실패코드).
Stage 3: 트레이싱(요청→CTO→SLM→Workspace 흐름).
Stage 4: 대시보드/알림 임계치.
Stage 5: 장애 대응 런북.
게이트: 장애 재현 없이도 “로그만 보고 원인 파악 가능”해야 완료.
4. 분야별 분리 전략(스파게티 방지)

domain: 순수 모델/규칙(외부 의존성 금지).
application: 유스케이스/오케스트레이션.
adapters: LLM, DB, 큐, API 구현체.
interfaces: CLI/API 엔드포인트.
observability: logger/metrics/tracing.
의존성 방향은 interfaces -> application -> domain, adapters -> application/domain만 허용.
5. 리팩터링 쉽게 만드는 규칙

모듈 간 통신은 “직접 참조”보다 “명시적 인터페이스 + 이벤트” 우선.
PR 단위는 작게(기능 1개, 책임 1개).
파일이 커지면 기능 단위로 즉시 분해.
테스트 없는 기능 병합 금지.
Deprecated -> Replace -> Remove 3단계로 변경.
6. 문제 위치를 빠르게 찾는 운영 규칙

모든 실행에 run_id 부여.
모든 작업에 task_id, 모든 메시지에 message_id 부여.
에러코드는 분류형으로 강제(E-LLM-TIMEOUT, E-PARSE-JSON, E-DEPS-BLOCKED).
실패는 단계별 대시보드에서 집계.
“단계 실패 시 되돌릴 지점”을 단계마다 1개씩 고정.
