## 작업 내용

### 1. `adapters/file_storage.py` 구현

`FileStorage` 클래스 신규 작성. TaskResult 목록을 받아 파일 시스템에 저장하고 README를 자동 생성하는 어댑터.

| 메서드 | 역할 |
|--------|------|
| `save_result_files(project_name, results, output_dir)` | `outputs/<slug>/` 경로에 성공한 태스크 파일만 저장 |
| `generate_readme(project_name, strategy, results)` | tech_stack, constraints, 파일 목록, 셋업 커맨드를 마크다운으로 조합 |
| `write_readme(project_name, strategy, results, output_dir)` | `README.md` 파일로 기록 후 경로 반환 |

프로젝트명은 `re.sub(r"[^\w-]", "-", name.lower())` 로 slug 변환 — 공백·특수문자가 포함된 이름을 안전한 디렉터리명으로 정규화.

### 2. `application/agent_factory.py` 구현

`SystemConfig` (dataclass) + `AgentFactory` 신규 작성. 컴포지션 루트 역할 — 구체 어댑터를 조립하여 에이전트 팀을 생성.

```python
@dataclass
class SystemConfig:
    cto_model: str = "llama3.1:8b"
    slm_model: str = "gemma4:e4b"
    mlops_model: str = "llama3.2:3b"
    ollama_base_url: str = "http://localhost:11434"
    output_dir: Path = Path("outputs")
    db_path: str = "data/compani.db"
    run_id: str = field(default_factory=lambda: uuid.uuid4().hex[:8])
```

`AgentFactory.create_cto(team=None)` 는 `Mapping[str, AgentPort]` 를 받도록 설계 — dict 불변성(invariance) 문제로 mypy 오류가 발생해 `Mapping`(공변) 타입으로 수정.

### 3. `main.py` 완성 (파이프라인 오케스트레이터)

플레이스홀더였던 `app_main` 을 완전한 파이프라인으로 교체.

```
orchestrate_project():
  1. CTO.create_strategy(request)
  2. CTO.decompose_tasks(strategy)
  3. asyncio.gather(*[team[role].execute_task(task) for task in tasks])  ← 병렬 실행
  4. FileStorage.save_result_files + write_readme
  5. ProjectResult 반환
```

각 태스크는 `asyncio.gather(..., return_exceptions=True)` 로 실행 — 하나가 실패해도 나머지를 계속 실행. `isinstance(r, TaskResult)` 로 성공 결과만 필터링.

`ProjectResult` 는 `project_name, success, total_tasks, completed_tasks, failed_tasks, output_dir, files_generated, duration_seconds` 포함.

## 결정 이유

### 왜 `asyncio.gather`로 태스크 병렬 실행했는가?

`BaseSLMAgent` 는 내부적으로 의존성 폴링(`_wait_dependencies`)을 구현하고 있어 각 에이전트가 스스로 대기한다. 따라서 오케스트레이터에서 레벨별 순서 제어 없이 `gather` 하나로 전부 동시 시작해도 의존성이 자동 처리됨. 레벨별 분리(Level 0 → Level 1 실행)는 코드 복잡도만 높이고 실익이 없어 단순하게 유지.

### 왜 `return_exceptions=True`를 사용했는가?

태스크 하나의 실패가 전체 파이프라인을 중단시키면 안 됨. 부분 성공을 허용하여 성공한 파일은 저장하고 `failed_tasks` 카운트에 반영. `CTO.review_progress` 가 50% 실패 시 ABORT를 결정하는 구조와 일치.

### 왜 `AgentFactory` 를 `application/`에 두었는가?

컴포지션 루트는 모든 계층을 알아야 하므로 아키텍처 엄격성과 충돌한다. `main.py` 에 두면 모듈이 비대해지고 테스트 주입이 어려워 `application/` 에 분리. 이 파일은 설계 규칙의 의도적 예외로 인정.
