## 3. 결정 이유

- **React→Flutter 트랙 분리**: React와 Flutter는 완전히 다른 기술 스택. 누적 검증 적용 시 Flutter 응답에서 `.tsx` 요건 등 React 요건이 강제되는 구조적 오류 발생. 두 트랙을 독립적으로 관리
- **Stage 4 프레임워크 강제 (`flutter`)**: LLM이 잘못된 프레임워크 선언을 했을 때 즉시 재시도하여 사용자에게 올바른 코드 전달
- **`const` 생성자 검증 (`const ` 단순 문자열 매칭)**: Flutter에서 `const`는 위젯 성능 최적화의 핵심. 전체 파일 대상 단순 검색으로 충분
- **전략 변경 없음**

## 4. 발생한 리스크

| 리스크 항목 | 심각도 | 발현 가능 시점 | 대응 방안 |
|------------|-------|--------------|----------|
| `const ` 단순 검색으로 `const String name = 'foo'` 같은 변수 선언도 통과 (const 생성자 아님) | C | Stage 4 이후 | `const MyWidget(` 패턴으로 강화 가능 |
| `include_stateless=False` 테스트 데이터가 `main.dart`까지 `StatefulWidget`으로 변경해야 했음 — 검증 키워드가 전체 파일에 걸쳐 체크되어 테스트 데이터 관리 복잡 | C | 유지보수 시 | widget 파일 경로 기준으로 검증 범위를 좁힐 수 있음 |
