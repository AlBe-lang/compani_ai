### 1-3. 핵심 로직 요약

**집계 (`_aggregate` in `emergency_meeting.py`)**
```python
tally = await self._weighted_tally(votes)      # collaboration*0.5 + precision*0.5, abstain 제외
cto_outcome = await self._invoke_cto_with_retry(...)  # 최대 cto_max_retries 회
if cto_outcome is not None:
    → DecisionSource.CTO, final_decision = cto의 decision
else:
    metrics.record_fallback(...)
    → DecisionSource.DNA_FALLBACK, final_decision = _resolve_fallback_winner(...)
```

**CTO 재시도 (`_invoke_cto_with_retry`)**
- 각 시도당 `asyncio.wait_for(timeout=30s)` + JSON 파싱 + MeetingDecision 검증
- 실패 사유 (TimeoutError / 파싱 실패 / 잘못된 decision) 셋 모두 재시도 대상 — `create_strategy`와 같은 기준
- 재시도 간격 기본 2초 × 실패 시도 수

**Tie-break (`_resolve_fallback_winner`)**
- 단일 최대값 → 그대로
- 동률 & blocking 소유자 vote가 동률군에 포함 → 소유자 vote 채택
- 동률 & 소유자 vote 동률군 밖 → 알파벳순 결정론적 선택 (테스트 안정성)
- 전원 abstain → `ESCALATE` (CTO에게 재계획 요청)

**스토리지 키 포맷**
- SQLite: `meeting:{meeting_id}` (고유 키, 16자 uuid hex)
- Qdrant 인덱싱: 기존 `task_results` 컬렉션에 `"[meeting] {reason} → {decision}: {cto_reason}"` 텍스트로 삽입, `success = (final_decision != ABORT)`

## 3. 결정 이유

### 3-1. 왜 MessageType 확장 대신 `context` dict 메타인가 (Q3 결정)
- `MessageType` enum 확장은 계약 변경 (Part 6 S3 AgentDNA genes처럼 BREAKING CHANGE 급). 합본 정비 후 Part 7 첫 Stage에서 또 BREAKING CHANGE는 피하고 싶음
- `Message.context: dict[str, object]`가 이미 자유 형식 — `{"meeting_id", "meeting_role", "possible_decisions"}` 메타 실으면 충분
- EventBus 기반 pub/sub(B안)은 현 `InProcessEventBus`가 메모리 기반이라 Rule 10 §1 위반 위험 → SQLite 기반 메시지 큐 유지가 정합

### 3-2. 왜 DNA 가중치 = collaboration×0.5 + precision×0.5인가
- 협업 회의 품질에 `collaboration` 유전자가 직접 관련
- 의견의 신뢰도에 `precision` 유전자가 관련
- 50/50 분할은 Stage 1의 가장 단순한 defensible prior. Phase 8 또는 Stage 2에서 역할별 가중치 튜닝 검토 가능

### 3-3. 왜 CTO 재시도 3회 × 2초 간격인가 (Q1 (a), (b) 결정)
- `cto_agent.py::create_strategy`가 이미 `DEFAULT_MAX_RETRIES = 3`, `DEFAULT_RETRY_DELAYS = (2.0, 4.0, 8.0)` 패턴. 본 회의는 간격을 단순화(고정 2초)해 구현 복잡도 감소
- 실패 기준 3종(타임아웃 / JSON 파싱 실패 / 잘못된 MeetingDecision enum) 모두 재시도 대상 — `_invoke_cto_with_retry` 루프 한 번으로 통합

### 3-4. 왜 폴백 발생을 `MetricsCollector.record_fallback`으로 기록하는가 (Q1 (d) 결정)
- 단순 카운터가 아닌 **run_id, component, reason, trace_id, recorded_at**까지 기록 → Phase 8 CEO 대시보드에서 "CTO 가용성 저하" 지표로 활용 가능
- `RunSummary.fallback_count`가 `get_run_summary()` 반환에 포함되어 현존 메트릭 API에서 바로 조회

### 3-5. 왜 tie-break가 "소유자 우선"인가 (Q1 (c) 결정)
- 동률 시점은 당사자 권한이 가장 합리적 — blocking을 감지한 주체가 원인을 가장 잘 알고 있다는 가정
- 소유자가 abstain이거나 동률군 밖 vote를 했다면 **알파벳순 deterministic** 선택 — 테스트 재현성 확보

### 3-6. 왜 `MeetingDecision`에 `ABSTAIN`을 넣지 않았는가
- 최종 결정 공간은 "시스템이 다음에 할 일"이어야 함 — RETRY/REROUTE/ESCALATE/ABORT 4가지가 실행 가능한 액션
- ABSTAIN은 개별 투표의 **메타 상태** 이지 결정이 아님 → `MeetingVote.abstained: bool` 플래그로 분리

### 3-7. 왜 `stage_gate.py`의 `emergency_meeting` 주입을 선택(Optional)으로 했는가
- Part 6 이전 배포 환경·테스트 환경에서 backward compat 필요
- 미주입 시 기존 `evaluate_emergency` 동작(고정 REPLAN) 그대로 — 회귀 없음
- 통합 테스트 `test_stage_gate_without_meeting_falls_back_to_fixed_replan`으로 회귀 검증

### 3-8. 왜 `meeting_participation_count`를 AgentDNA의 새 필드로 넣었는가 (Q5 결정)
- `genes` dict에 숫자 카운터를 섞으면 의미 충돌 (genes는 `[0, 1]` EMA 유전자)
- 별도 int 필드로 분리 → 용도 명확. Stage 2에서 `collaboration` EMA 반영 시 "참여했지만 기권했다" 등을 카운터 기반으로 계산 가능
