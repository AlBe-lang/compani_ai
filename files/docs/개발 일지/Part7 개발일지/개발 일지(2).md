## 1. 작업 내용

### 1-1. 배경

Stage 1에서 `blocking.detected` → 긴급 미니 회의 경로를 구축했다. Stage 2는 **정상 경로에서의 자율 협업** — Task가 DONE 상태에 도달한 순간 **다른 역할 에이전트가 자동으로 코드 리뷰**를 수행하도록 확장.

Rule 9.1 §1·§2 기반 확정 결정:

| 항목 | 결정 |
|------|------|
| `task.completed` 이벤트 발행 주체 | **(A) 순수** — `SharedWorkspace.attach_result` 말미. 기존 `blocking.detected` 패턴과 동형 |
| 리뷰어 선정 | **(D) 하이브리드** — 고정 매핑 기본, `KnowledgeGraph.find_best_responder`가 다른 역할을 반환하면 opportunistic 사용. pluggable `ReviewerSelector` Protocol |
| 리뷰 결과 처리 | **severity 기반 하이브리드** — APPROVE/REJECT는 상태 전이, MAJOR/CRITICAL은 `pending_rework=True` 플래그만 기록(실제 재실행은 Stage 3) |
| 계약 | **(A) 신규 `PeerReviewResult`** — 기존 `ReviewResult`(Gate 레벨)와 의미 분리 |
| 트리거 | **`PeerReviewMode` enum(OFF/ALL/CRITICAL)** — 기본 OFF. VRAM이 아닌 "감내 가능한 추가 실행 시간" 기준으로 사용자 선택 |

### 1-4. 핵심 로직 요약

**이벤트 흐름**
```
BaseSLMAgent.execute_task 성공
  └─ workspace.set_status(DONE)               — "work_item.updated" 발행
  └─ workspace.attach_result(work_item_id, result)
       └─ item.status == DONE 이면 "task.completed" 발행  ← Stage 2 신규
            └─ PeerReviewCoordinator._on_task_completed
                 └─ asyncio.create_task(self.review_task(payload))
                      └─ _should_review → _select_participants → _perform_review (LLM)
                         → _persist(SQLite + Qdrant 선택) → _apply_state_transition → _feedback_dna
```

**리뷰어 선정 (`FixedWithKGFallbackSelector`)**
```
고정 매핑: backend→frontend, frontend→backend, mlops→backend
KG opportunistic:
  find_best_responder(approach) ≠ None AND ≠ author_role AND in _KNOWN_ROLES
  → KG 결과 채택. 그 외 → 고정 매핑
```

**상태 전이 (`_apply_state_transition`)**
```
APPROVE                    → DONE 유지, workspace 터치 안 함
REQUEST_CHANGES (MINOR)    → DONE 유지, pending_rework=False
REQUEST_CHANGES (MAJOR)    → DONE 유지, pending_rework=True (Stage 3 재실행 대상)
REQUEST_CHANGES (CRITICAL) → DONE 유지, pending_rework=True (Stage 3 재실행 대상)
REJECT                     → workspace.set_status(FAILED) 호출
```

**DNA 피드백 (`update_review_feedback`)** — Stage 1 Q5 C 유보 항목 해소
```
Reviewer:
  review_count += 1
  genes['collaboration'] EMA update with sample=1.0

Author:
  genes['precision']    EMA update with sample = {APPROVE: 1.0, REQUEST_CHANGES: 0.5, REJECT: 0.0}
  genes['code_quality'] EMA update with sample = {APPROVE: 1.0,
                                                   REQUEST_CHANGES MINOR: 0.8, MAJOR: 0.5, CRITICAL: 0.2,
                                                   REJECT: 0.0}
```

**트리거 정책 (`_should_review`)**
```
OFF       → return False
ALL       → return True
CRITICAL  → has_dependencies OR duration_sec ≥ critical_duration_sec
```

## 3. 결정 이유

### 3-1. 왜 `attach_result`에서 `task.completed`를 발행하는가 (Q1)
- 기존 `SharedWorkspace.set_status(BLOCKED)`이 `blocking.detected`를 발행하는 패턴과 동형
- `attach_result`는 Task당 정확히 한 번 호출되고 WorkItem + TaskResult 둘 다 접근 가능
- `BaseSLMAgent`에서 발행하면 4개 에이전트가 동일 로직을 복제 (DRY 위반) + EventBus 의존 관심사 유입
- 별도 래퍼(`TaskLifecycleCoordinator`)는 Rule 9.1 P-04 범위 확장

### 3-2. 왜 `ReviewerSelector` Protocol을 뒀는가 (Q2)
- Stage 3 작업("자동 리뷰어 선정")이 DNA 가중 / 부하 분산 / 이해 충돌 방지를 요구
- Coordinator 본체를 건드리지 않고 전략만 갈아끼우기 가능해야 Stage 3가 깔끔
- Rule 10 §3(교체 가능 Port) 정신 엄수
- Stage 2에서는 `FixedWithKGFallbackSelector` 하나만 제공 — 의존성은 KG만

### 3-3. 왜 MAJOR/CRITICAL도 상태 전이 대신 `pending_rework` 플래그인가 (Q3)
- 사용자 제안은 "CRITICAL → IN_PROGRESS 복귀". 하지만 Stage 2에서 **재실행 엔진이 없음** — 에이전트를 자동 호출하는 체계는 Stage 3에서 구축됨
- 상태만 IN_PROGRESS로 바꾸고 아무도 실행하지 않으면 StageGate 평가가 "진행 중으로 멈춘 Task"를 과도하게 보고
- 따라서 Stage 2는 "재작업 필요 시그널"까지만 기록 — `pending_rework=True`. Stage 3의 rework scheduler가 이 플래그를 읽어 실제 status 전이 + 재실행 담당
- 이로써 사용자 제안의 하이브리드 정신(severity 기반)은 유지하면서 WorkItem 계약 무변경

### 3-4. 왜 `REJECT`만 `set_status(FAILED)` 호출하는가
- REJECT = "회복 불가, 폐기" — WorkItem 자체가 실패한 것과 의미가 같음
- REQUEST_CHANGES(어느 severity든)는 "수정 가능" 의미 → DONE 유지가 자연스러움
- FAILED 전이가 실패하는 경우(이미 다른 상태로 전이됨 등)를 대비해 예외 비치명적 처리(로그만)

### 3-5. 왜 Coordinator가 MessageQueue가 아닌 LLM 직접 호출인가
- Stage 1 `EmergencyMeeting`은 MessageQueue를 사용했지만 실제로는 참여자 에이전트가 메시지를 수신·응답하는 메시지 루프가 없어 pre-populated inbox로만 테스트
- 피어 리뷰는 "리뷰어 역할의 모델로 LLM을 한 번 호출"하는 것이 본질 — 메시지 왕복을 거칠 이유가 적음
- `StageGateMeeting._delegate_to_cto`가 CTO를 직접 호출하는 것과 같은 패턴 — 일관성
- MessageQueue 활용이 필요한 경우는 Stage 3에서 "리뷰 알림 감사 로그" 용도로 선택적 추가

### 3-6. 왜 DNA 샘플값을 이렇게 테이블화했는가 (§1-4)
- APPROVE=1.0 / REJECT=0.0 은 이견 없는 경계값
- REQUEST_CHANGES 전체를 0.5로 둔 것은 "코드가 작동하나 개선 필요" 상태 = 중립
- code_quality는 severity를 반영해 MINOR(0.8) / MAJOR(0.5) / CRITICAL(0.2)로 세분화 — 현실적 리뷰 어투와 대응
- 모듈 수준 함수로 노출해 테스트에서 EMA 식 검증 없이 직접 샘플값 검증 가능

### 3-7. 왜 기본값이 `PeerReviewMode.OFF`인가 (Q5)
- Part 7 Stage 2에서 도입되지만 기본 활성은 보수적 접근 — 기존 runtime의 예상 행태 변경을 최소화
- 사용자가 의도적으로 CRITICAL 또는 ALL 설정해야 활성
- CEO 대시보드(Part 8)에서 runtime 토글로 바꿀 수 있는 자리를 마련

### 3-8. 왜 VRAM이 아닌 "감내 가능한 추가 시간"이 가이드인가 (사용자 질문 반영)
- 피어 리뷰는 **새 모델을 메모리에 올리지 않음** — 이미 로드된 reviewer 역할 모델을 순차 재호출
- 추가 VRAM 요구 = 0, 추가 시간 = 약 2배(ALL 모드)
- UI가 생길 때 "GPU VRAM 8GB 이상" 같은 문구는 오해 유발 — 대신 "프로젝트당 추가 대기 시간 감내 가능하면" 기준 제시
