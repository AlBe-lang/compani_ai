### 1-4. 핵심 로직 요약

**DNAAwareSelector 공식**
```
collab  = DNA.genes["collaboration"]   # [0,1]
prec    = DNA.genes["precision"]       # [0,1]
recent  = 현재 run 내 리뷰 카운트(self._recent_counts[role])

base    = 0.5 × collab + 0.5 × prec    # [0,1]
decay   = exp(-recent / 5.0)           # (0,1]
score   = base × decay                 # (0,1]
```

**COI 적용 순서**
```
1. 후보 풀: backend, frontend, mlops
2. Tier 1 exclude: 작성자 role + dep_source_roles
3. Tier 2 penalty: role ∈ transitive_roles 면 score × 0.7
4. argmax (동률 시 role asc), score == 0.0 이면 None
5. None 이면 호출자가 FixedWithKGFallbackSelector 로 cascade (PeerReviewCoordinator 가 담당)
```

**Rework 흐름 (enabled=True 일 때)**
```
Peer review 끝
  pending_rework=True?
    └─ EventBus.publish("review.rework_requested", payload)

ReworkScheduler._on_rework_requested
  └─ create_task(handle_rework(payload))

handle_rework:
  1. WorkItem 찾기
  2. rework_count >= max_attempts ? → set_status(FAILED) + record_fallback → return False
  3. workspace.reopen(work_item_id, reason=peer_review:SEVERITY)
     → DONE→IN_PROGRESS, rework_count+1, work_item.reopened 이벤트
  4. 원 Task 재구성 (task_result_snapshot 기반, dependencies 는 빈 리스트로)
  5. agent.execute_task(task, context={"review_feedback":...}) 비동기 fire-and-forget
  6. execute_task 는 work_item_id 를 context 로 받아 기존 WorkItem 재사용, LLM 호출
  7. 성공 시 attach_result → 다시 task.completed → 리뷰 루프 재진입
  8. max_attempts 도달 시 자연스럽게 FAILED 로 수렴
```

**KG expertise 영속화**
```
_update_expertise_ema 호출 시:
  1. 인메모리 EMA 업데이트 (동기)
  2. storage 주입됐으면 asyncio.create_task 로 _persist_expertise 비동기 실행
  3. persist 실패 시 warn 로그만, 인메모리 값은 유지 (self-heal)

load_expertise (시작 시 명시 호출):
  1. storage.query() 전체 로드
  2. _kind == "kg_expertise" row 만 추출
  3. self._expertise[role][topic] = ema_value
```

## 3. 결정 이유

### 3-1. 왜 승법 감쇠 (Q1 답의 심화)
- **선형 (A)**: `recent_review_count` 가 충분히 크면 score가 음수가 되어 "유능하지만 한가한" 기대와 반대 결과 가능
- **Softmax (C)**: 비결정적 → 테스트 재현성 위해 seed 고정 필요. 초기 DNA가 전부 0.5 근처인 현 시스템에선 탐험 가치 낮음
- **승법 감쇠 (B)**: 결정론적, score 범위가 항상 `(0, 1]`, 지수 감쇠가 자연스럽게 부하 분산, 튜닝 파라미터 하나(`k`)
- **k=5 선택**: recent=5 에서 decay≈0.37(→ 동등 DNA 대비 저부하 에이전트가 선호됨), recent=10 에서 ≈0.14(강하게 억제)

### 3-2. 왜 Tier 1 + Tier 2 하이브리드 (Q2 답의 심화)
- 3-role 시스템에서 Tier 2 까지 hard exclude하면 후보 0명 발생 → 시스템 정지. **hard vs soft 구분 필수**
- 직접 의존성(Tier 1): 이해관계가 명백 → 배제
- Transitive(Tier 2): 정보 격리 요구는 있으나 점수 감점으로 충분 (×0.7)
- 후보 0명 시 `FixedWithKGFallbackSelector` cascade — **시스템 멈추지 않음**이 최우선

### 3-3. 왜 `reopen()` 메서드 + `rework_count` 필드 (Q3 답의 심화)
- `_ALLOWED_TRANSITIONS` 에 `(DONE, IN_PROGRESS)` 추가하면 기존 가정("DONE 은 불변")이 깨질 수 있어 위험
- 명시적 `reopen()` 메서드는 의도를 함수명에 박아 넣어 감사 추적 쉽고 이벤트 발행 자연스러움
- `rework_count` 는 non-breaking(기본 0) 필드로, 재실행 상한 강제의 기반 카운터

### 3-4. 왜 EventBus 이벤트 트리거 (Q4 답의 심화)
- `blocking.detected`(Part 6 S3), `task.completed`(Stage 2) 와 일관 — 새 메커니즘 도입 없이 기존 패턴 연장
- 즉시성: 리뷰 직후 재실행 스케줄 → 사용자 체감 지연 최소
- 무한 루프 방지: `WorkItem.rework_count >= max_attempts` → FAILED 강제, 자연스럽게 탈출
- 동시성: `asyncio.create_task` 로 독립 스케줄링 (Stage 1/2 검증된 패턴)

### 3-5. 왜 DNAManager 대칭 영속화 (Q5 답의 심화)
- DNAManager 는 매 update 마다 `_save` 호출 — KG `_update_expertise_ema` 도 같은 패턴이면 두 모듈의 멘탈 모델 단일화
- 빈도 낮음: Q&A/Task 완료 당 1회. SQLite WAL 로 충분
- fire-and-forget: 메인 경로 block 안 함. 크래시 시 마지막 몇 개 손실이지만 EMA self-heal (Rule 10 §1 허용 범위)
- 별도 테이블(`005_expertise.sql`)은 선언적 — 실제 저장은 kv_store 의 `kg_expertise:{role}:{topic}` 키로. 004_agent_dna.sql 과 동일 패턴

### 3-6. 왜 rework 시 `_handle_dependencies` 를 스킵하는가
- reopen 직후 WorkItem 은 이미 IN_PROGRESS 상태 → `_handle_dependencies` 가 `set_status(IN_PROGRESS)` 호출하면 `(IN_PROGRESS, IN_PROGRESS)` 전이로 AdapterError
- 또한 rework 는 "이미 완료했던 task 재실행" 이므로 의존성 대기 재검증 불필요 (초기 실행 시 이미 통과)
- `context["work_item_id"]` 존재를 is_rework 판정 키로 사용
