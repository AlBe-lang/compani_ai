# 개발 일지 (4) — 에이전트별 독립 작업 계획

> 기준일: 2026-02-21
> 이전 문서: 개발 일지(3) — 파트별 세부 작업 계획 (구조 미흡으로 본 문서로 대체)
> 변경 이유: 에이전트별 독립 구현 + 단계적 성능 고도화 구조로 재설계

---

## 설계 원칙

```
1. 에이전트 독립성
   - 각 에이전트(CTO / Backend / Frontend / MLOps)는 별도 Part로 분리
   - 개발 중 타 에이전트 구현체를 직접 import하지 않음
   - 모든 통신은 domain/ports.py의 Protocol(인터페이스)만 사용

2. 단계적 성능 고도화
   - 각 에이전트 Stage 1 = 기본 동작 보장
   - Stage N+1 = 이전 Stage의 품질을 개선 (추가 기능 X, 품질 향상 O)
   - 고도화는 협업 레이어 통합 이후에 진행

3. 스파게티 방지
   - 에이전트 → 에이전트 직접 참조 금지
   - 에이전트는 WorkSpacePort / MessageQueuePort만 알고 있음
   - 테스트 시 MockWorkSpacePort / MockMessageQueuePort 사용

4. 개발 순서
   Part 0 (기반) → Part 1 (CTO) → Part 2 (Backend) →
   Part 3 (Frontend) → Part 4 (MLOps) → Part 5 (협업 통합) →
   Part 6 (고도화) → Part 7 (CEO 대시보드)
```

---

### Stage 1: 전략 수립 (create_strategy)

**목적**: 사용자의 자연어 요청을 받아 Strategy 객체를 생성한다.
LLM이 JSON을 반환하지 못할 경우 재시도 3회 후 E_PARSE_JSON.

**선행 조건**: Part 0 Gate 통과

**세부 작업**:

```
1. application/cto_agent.py — CTOAgent 클래스 기본 구조
   - __init__(self, llm: LLMProvider, run_id: str)
   - create_strategy(project_request: str) -> Strategy

2. application/prompts/cto/strategy.txt
   시스템 프롬프트: "당신은 시니어 CTO입니다..."
   출력 형식: JSON Schema 명시 (project_name / description / tech_stack / constraints)

3. JSON 파싱 유틸리티 (observability/parsers.py)
   - parse_json_response(text: str) -> dict
   - 코드 블록(```json ... ```) 내 JSON 추출 포함
   - 실패 시 E_PARSE_JSON 발생

4. 재시도 데코레이터 (최대 3회)

5. tests/unit/test_cto_strategy.py
   - MockLLMProvider로 정상 응답 테스트
   - JSON 파싱 실패 → 재시도 → 성공 테스트
   - 3회 모두 실패 → E_PARSE_JSON 테스트
```

**생성 파일**:

```

├── cto_agent.py
└── prompts/
    └── cto/
        └── strategy.txt

observability/
└── parsers.py

tests/unit/
└── test_cto_strategy.py
```
