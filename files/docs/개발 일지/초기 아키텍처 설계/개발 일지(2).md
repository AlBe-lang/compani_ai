## 핵심 원칙 (변경 없음)

핵심은 **Part(파이프라인) → Stage(단계) → Domain(분야)** 3계층으로 운영하는 것입니다.
정합성을 먼저 맞추고, 파트를 순차 실행하는 방식이 가장 안전합니다.
**안정과 완성형**을 최우선으로 하며, 각 파트는 Gate 조건을 통과해야만 다음 파트로 진입합니다.

## 0. 심각도 분류 기준 (추가)

모든 이슈와 리스크는 아래 4축으로 평가하고, **4축 중 가장 높은 등급**을 최종 심각도로 결정합니다.

### 4축 평가표

| 축 | S (즉시 위험) | A (고위험) | B (중위험) | C (저위험) |
|----|-------------|-----------|-----------|-----------|
| **차단성** — 진행을 막는가 | 해당 단계 실행 자체 불가 | 완료 기준 충족 불가 | 진행되나 다음 단계 품질 저하 | 진행 가능, 기술 부채만 발생 |
| **전파성** — 얼마나 넓게 퍼지는가 | 모든 Part에 영향 | 1개 Part 전체에 영향 | 특정 Stage에만 영향 | 특정 파일·모듈에만 영향 |
| **발현 시점** — 언제 터지는가 | Part 1에서 즉시 | Part 2 진입 직후 | Part 2 중반 이후 | Part 3 이후 |
| **수정 비용** — 나중에 고치면 얼마나 비싼가 | 아키텍처 재설계 | 여러 파일·모듈 동시 수정 | 단일 컴포넌트 수정 | 설정·문서 수정으로 해결 |

### 심각도별 처리 원칙

| 등급 | 의미 | 처리 원칙 |
|------|------|---------|
| **S** | 즉시 위험 | 다음 Stage 착수 전 반드시 해결. 미해결 시 Gate 통과 불가 |
| **A** | 고위험 | 해당 Part 내에서 해결. Stage 계획에 명시적으로 포함 |
| **B** | 중위험 | 해당 Part 종료 전 해결. 단, 발현 시점이 S이면 선행 처리 |
| **C** | 저위험 | 기술 부채로 관리. 다음 Part 시작 전 일괄 정리 |

#### Part 1 Gate 조건 ← **수정됨**

개발 일지(1)의 Gate는 CI Gate 순환 문제가 있었습니다. 다음과 같이 2단계로 분리합니다.

```
[내부 완료 기준 — Stage 1~4 도달 시]
  "로컬 make test-fast 전 구간 green"
  "CI yml 파일 존재 및 문법 검증 통과"

[Part 1 → Part 2 진입 Gate]
  "main 브랜치 첫 PR이 CI에서 green 상태로 머지 완료"
```

> **이전 Gate 문제**: "PR에서 자동 품질 체크 100% 통과"가 Stage 4(CI 구축) 완료를 전제로 하면서, 동시에 Stage 4의 완료 기준이기도 했음 → 자기 참조 순환.

## 4. 분야별 분리 전략 — 스파게티 방지 (변경 없음)

```
domain/        순수 모델·규칙  (외부 의존성 금지)
application/   유스케이스·오케스트레이션
```

```
interfaces  →  application  →  domain
adapters    →  application  →  domain
observability → 모든 레이어 (단방향 수집만)
```

- 테스트 없는 기능 병합 금지

## 6. 문제 위치를 빠르게 찾는 운영 규칙 (변경 없음)
