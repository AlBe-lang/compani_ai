# Part2 개발 일지 (1) — Stage 1: 기본 FastAPI 코드 생성 에이전트 착수
> 작성일: 2026-02-23 | Part: 2 | Stage: 1

## 1. 작업 내용
Part 1 Gate 완료 상태를 기준으로 Part 2 Stage 1 범위만 착수했습니다.  
`ports.py`는 변경하지 않고, Agent 내부에서 `WorkItem`을 생성해 `register(work_item)`을 호출하도록 구현했습니다.  
`BaseSLMAgent + BackendSLMAgent` 구조를 채택해 Part 3/4 확장 시 클래스 분리를 유지할 수 있게 했습니다.

## 2. 결과물
생성 파일:
- `src/application/base_agent.py` — 공통 실행 플로우(의존성 대기, Q&A fallback, 파싱 재시도, 실패 마킹)
- `src/application/backend_agent.py` — Backend Stage 1 에이전트 구현체
- `src/application/prompts/backend/stage1_basic.txt` — Stage 1 백엔드 프롬프트
- `src/application/prompts/backend/__init__.py`
- `tests/unit/test_backend_stage1.py` — Stage 1 단위 테스트 4개

수정 파일:
- `src/application/__init__.py` — 신규 에이전트/설정 export 추가

테스트 시나리오(4개):
1. 성공 경로 + 의존성 대기 상태 전이(`WAITING -> IN_PROGRESS -> DONE`)
2. 의존성 타임아웃 시 `E_DEPS_TIMEOUT`
3. Q&A 타임아웃 시 빈 응답 fallback 후 계속 진행
4. `files` 스키마 불완전 응답에서 재시도 후 성공

## 3. 결정 이유
`BaseSLMAgent + BackendSLMAgent`는 Stage별 프롬프트 고도화와 역할별 분리를 동시에 만족합니다.  
`WorkSpacePort.register` 시그니처는 Part 0/1에서 확정된 계약이므로 유지하고, 내부 생성 전략으로 규칙 문서 흐름과 양립시켰습니다.  
Stage 1 범위에서 통합/E2E는 제외하고 단위 테스트 중심으로 검증 지점을 명확히 했습니다.

## 4. 발생한 리스크
| 리스크 항목 | 심각도(S/A/B/C) | 발현 가능 시점 | 대응 방안 |
