# Part2 개발 일지 (4) — Stage 3: 에러 처리 + 입력 검증 생성
> 작성일: 2026-04-21 | Part: 2 | Stage: 3

---

## 1. 작업 내용
Part 2 Stage 3 범위만 고정해 Backend 에이전트의 에러 처리·입력 검증 패턴 생성 능력을 추가했습니다.  
`BackendSLMConfig.stage=3` 경로에서 `stage3_validation.txt` 프롬프트를 선택하도록 구현했습니다.  
`_validate_stage3_files()`를 추가해 생성 결과에 `HTTPException` 사용 여부와 `ErrorResponse` 스키마 파일 존재 여부를 검증합니다.  
Stage 2 검증(`_validate_stage2_files`)은 유지한 채 Stage 3 검증을 누적 적용하는 구조로, 낮은 단계의 품질 기준이 높은 단계에서도 보장됩니다.

## 2. 결과물
생성 파일:
- `src/application/prompts/backend/stage3_validation.txt` — Stage 3 백엔드 프롬프트 (HTTPException, ErrorResponse 규칙 포함)
- `tests/unit/test_backend_stage3.py` — Stage 3 단위 테스트 4개

수정 파일:
- `src/application/backend_agent.py` — stage 3 프롬프트 매핑 및 `_validate_stage3_files()` 추가

테스트 시나리오(4개):
1. Stage 3 성공 경로: HTTPException + ErrorResponse 포함 응답에서 결과 반환 및 프롬프트 선택 검증
2. HTTPException 누락 시 재시도 후 성공
3. ErrorResponse 스키마 파일 누락 시 재시도 후 성공
4. Stage 1/2 회귀: Stage 3 추가 후에도 stage=1, stage=2 경로가 정상 동작함을 검증

## 3. 결정 이유
Stage 2와 동일한 누적 검증 전략을 유지했습니다. `super()._validate_files()` 호출 후 단계별 추가 검증을 적용해, 낮은 단계 기준을 재정의하지 않고 확장합니다.  
HTTPException은 내용(content) 기반 검색, ErrorResponse 스키마는 경로 패턴(`/schemas/*.py`) + 내용(content) 기반으로 이중 검증해 오탐을 줄였습니다.  
전략 문서(개발 일지(4).md) Part 2 Stage 3의 "에러 처리 + 입력 검증 추가" 정의를 그대로 따랐습니다.  
전략 변경 이력: 없음.
