# 이전 (버그)
def _on_blocking_detected(self, payload: dict[str, object]) -> None: ...

## 3. 결정 이유

### 3-1. 왜 E501은 `# noqa`로 처리했는가
해당 8개 라인은 **테스트 고정 문자열**(Dockerfile, tsx content, CI YAML 등) 또는 **에러 메시지 f-string**. 줄바꿈하면:
- 문자열 리터럴: escape sequence 분리로 가독성 급락
- f-string: 여러 줄로 분리해도 내용 파악이 더 어려움

대안 검토:
- **(A) black 재실행**: black의 line-length=100 이미 적용됨. 문자열 내부 개행은 black이 손대지 않음
- **(B) 줄바꿈**: 테스트 fixture는 "실제 출력 문자열"을 재현하는 성격이라 분해 시 검증 의도 모호
- **(C) `# noqa: E501`**: 명시적 예외. 코드 리뷰에서 "왜 예외인지" 즉시 판단 가능

C 채택. 프로덕션 코드는 모두 black으로 해결됐고, 예외는 테스트 fixture 8곳에 국한.

### 3-2. 왜 `_on_blocking_detected`에 기본값을 주었는가
시그니처를 `(event_type, payload)`로만 바꾸면 테스트의 직접 호출(`gate._on_blocking_detected(payload)`)이 모두 깨진다. 두 매개변수 모두 기본값을 주면:
1. EventHandler 프로토콜과 구조적 호환 유지
2. 테스트는 `_on_blocking_detected("blocking.detected", payload)` 형태로 **명시 호출**하여 의미 있는 호출
3. None 체크로 기본값 호출(`_on_blocking_detected()`) 안전하게 무시

프로토콜 준수 + 테스트 생존 + 방어적 처리 세 가지 동시 달성.

### 3-3. 왜 RedisCache 테스트에서 str → dict 래핑인가
`StorageValue: TypeAlias = dict[str, object]` (domain/ports.py:17)이 공개 계약. 이전 테스트 `cache.set("str_key", "hello")`는 계약 위반 호출이었지만 런타임에 JSON 직렬화로 우연히 동작하고 있었음. 두 선택:
- **(A) 계약을 넓힘**: `StorageValue = dict | str | int | ...`로 변경 → 연쇄 수정 대규모, Rule 9.1 P-04 위반
- **(B) 테스트를 계약에 맞춤**: `cache.set("k", {"value": "hello"})`로 래핑

B가 정합성 있는 선택. 계약은 유지, 테스트 의도(string도 캐시 가능)는 dict 래핑으로 보존.

### 3-4. 왜 mock `call_count`/`called`에 `type: ignore`를 붙였는가
`manager._storage.load`는 `StoragePort.load` 타입(`Callable[..., Coroutine[...]]`)으로 추론되어 mypy는 `.call_count` 속성을 모른다. `MagicMock`으로 cast하면 보이지만, mock 조회를 위해 전역 cast 도입은 과함. `# type: ignore[attr-defined]`가 3줄의 단순 주석으로 끝나고, 테스트 의도는 명확 유지 — 가장 경제적.
