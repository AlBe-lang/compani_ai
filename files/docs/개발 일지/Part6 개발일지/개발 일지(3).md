### 1-2. `DNAManager` 구현 (`application/dna_manager.py`)

에이전트 DNA를 로드/업데이트/저장하는 Application 서비스.

**핵심 알고리즘 — EMA 업데이트** (α = 0.2):
```
new_gene = α × sample + (1 - α) × prev_gene
```
- `precision`: success 시 sample=1.0, 실패 시 0.0
- `code_quality`: success AND files>0 → 1.0, 그 외 0.0
- `debugging_skill`: error_code 없으면 1.0, 에러 코드 있으면 0.0
- `speed`: `max(0, 1 - duration_sec / 120)` — 120초 기준으로 정규화

### 1-3. `StageGateMeeting` 구현 (`application/stage_gate.py`)

**하이브리드 평가 전략** (규칙 기반 우선 → CTO 위임):

```
evaluate(work_items)
  → _check_numeric_gate()    # LLM 호출 없음, O(n)
      PASS → 즉시 반환
      FAIL → _delegate_to_cto()  # CTO.review_progress() 호출
                → ReviewDecision.REPLAN → GateVerdict.REPLAN
                → ReviewDecision.ABORT  → GateVerdict.ABORT
```

**설계 결정**: 규칙 기반만으로 갈지, LLM 위임만으로 갈지 검토. 실제 동작 가능한 수준을 기준으로 하이브리드가 최선:
- PASS 케이스(전체의 70~80%)는 LLM 호출 없이 즉시 처리 → 비용/속도 최적
- FAIL 케이스만 CTO에 위임 → 맥락 이해가 필요한 판단을 LLM이 처리

**긴급 Gate** (`evaluate_emergency`): `blocking.detected` 이벤트 수신 시 단일 아이템 기준으로 REPLAN 결정. 전체 평가와 별개로 실시간 대응.

**Gate 기준값 주입**: `GateConfig(max_failure_rate=0.3, max_avg_duration=120.0)`을 `SystemConfig`를 통해 외부에서 조정 가능. 코드 수정 없이 프로젝트별 임계값 변경.

### 1-4. `SharedWorkspace` — `blocking.detected` 이벤트 발행 (`adapters/shared_workspace.py`)

WorkItem 상태가 `BLOCKED`로 전이될 때 EventBus에 이벤트 게시:

```python
if status is WorkStatus.BLOCKED:
    await self._event_bus.publish(
        "blocking.detected",
        {"item_id": work_item_id, "agent_id": item.agent_id},
    )
```

`StageGateMeeting` 생성자에서 `event_bus.subscribe("blocking.detected", self._on_blocking_detected)`로 등록. EventBus 핸들러는 동기이므로 `asyncio.get_running_loop().create_task()`로 긴급 Gate를 비동기 소집.

## 3. 설계 원칙 및 트레이드오프

### 3-1. EMA vs 단순 평균

| 지표 | 방식 | 이유 |
|------|------|------|
| genes (precision 등) | EMA (α=0.2) | 최근 작업에 더 많은 가중치, 점진적 변화 |
| success_rate | 롤링 평균 | 전체 성공률은 해석 가능한 스칼라가 필요 |
| avg_duration | 롤링 평균 | 이상치 민감도 낮추고 전체 추세 파악 |

### 3-2. 긴급 Gate의 REPLAN 고정

`evaluate_emergency()`는 항상 `REPLAN`을 반환한다. ABORT를 바로 내리지 않는 이유:
- 단일 blocking은 재시도로 해결 가능한 일시적 상태일 수 있음
- 최악의 경우에도 CTO가 다음 일반 Gate에서 ABORT 결정 가능
- 즉각 ABORT는 완료된 성공 작업까지 버리는 비가역적 결과

### 3-3. `TYPE_CHECKING` 패턴

`BaseSLMAgent → DNAManager → AgentDNA` 의존 체인에서 DNAManager가 application 레이어에 있고 BaseSLMAgent도 application 레이어이므로 런타임 import가 순환을 야기할 수 있다. `TYPE_CHECKING` 블록 내 import로 타입 힌트는 유지하되 런타임 의존성은 제거.
