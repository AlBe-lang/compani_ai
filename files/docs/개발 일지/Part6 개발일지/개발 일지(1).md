기존 단위 테스트들이 `task.dependencies = [work_item_id]`로 작성되어 있었음 → `task.dependencies = [task_id]` 로 수정.

### 1-2. `MetricsCollector` (`observability/metrics.py`)

태스크 단위 실행 지표를 인메모리로 수집하고 SQLite에 플러시하는 컴포넌트.

| 메서드 | 역할 |
|--------|------|
| `record_task(run_id, task_id, agent_id, *, success, duration_sec, retries)` | `TaskMetric` 생성 + 현재 `trace_id` 자동 첨부 |
| `get_run_summary(run_id)` | `RunSummary` (성공/실패 카운트, 평균 소요 시간, 총 재시도) 반환 |

`RunSummary`는 `total_tasks`, `success_count`, `fail_count`, `avg_duration_sec`, `total_retries` 포함.

### 1-3. `TraceContext` (`observability/tracing.py`)

`contextvars.ContextVar`를 활용해 비동기 컨텍스트에 trace_id를 전파.

- `new_trace()` → 16자 hex trace_id 생성 후 컨텍스트에 설정, id 반환
- `get_trace_id()` → 현재 컨텍스트의 trace_id 조회 (미설정 시 `""`)
- `set_trace_id(trace_id)` → 기존 trace_id 이어받기 (분산 추적 용도)

`MetricsCollector.record_task()`는 기록 시 `get_trace_id()`를 자동으로 첨부해 메트릭과 로그의 연관성을 보장한다.

### 1-4. Logger stderr 리다이렉션 (`observability/logger.py`)

`structlog.PrintLoggerFactory(file=sys.stderr)` → `file=_DynamicStderr()` 로 교체.

`_DynamicStderr` 프록시 클래스는 write 시점에 `sys.stderr`를 조회하므로, pytest가 테스트마다 `sys.stderr`를 교체해도 "I/O operation on closed file" 오류가 발생하지 않는다.

Rich Live는 stdout을 사용, structlog는 stderr를 사용하므로 대시보드와 로그가 충돌하지 않는다.

## 3. 결정 이유

### 왜 `get_by_task_id`를 `WorkSpacePort` 프로토콜에 추가했는가?

`SharedWorkspace.get()` 내부에서 task_id 폴백 로직을 숨기는 방법도 있었다. 하지만 이는 `get(id)` 가 두 가지 의미(work_item_id와 task_id)를 가지게 되어 의도가 모호해진다. 프로토콜에 명시적으로 분리된 메서드를 추가하는 것이 호출부 코드의 가독성과 테스트 가능성 측면에서 우월하다.

### 왜 structlog를 stderr로 변경했는가?

Rich Live는 stdout을 통해 인터랙티브 UI를 렌더링하고, ANSI 이스케이프 시퀀스로 화면을 제어한다. structlog가 같은 stdout에 JSON 라인을 기록하면 화면이 깨진다. stderr 분리는 Unix 파이프라인 철학과도 일치한다(데이터는 stdout, 진단 정보는 stderr).

### 왜 `_DynamicStderr` 프록시가 필요했는가?

`PrintLoggerFactory(file=sys.stderr)`는 구성 시점의 `sys.stderr` 객체를 캡처한다. pytest가 테스트마다 `sys.stderr`를 교체하면 이전 참조는 닫혀 "I/O operation on closed file"이 발생한다. 프록시는 write 시점에 현재 `sys.stderr`를 조회함으로써 테스트 격리 문제를 해결한다.

### 전략 변경 이력

이전 세션에서의 전략 제안 중 두 가지가 미래지향적 관점에서 단기적이었음이 확인됨:
1. **EMA 미적용 → EMA(α=0.2) 채택**: 단순 대체는 학습 이력을 손실함. Part 6 Stage 3에서 DNAManager 구현 시 반영.
2. **Qdrant 불필요 → Qdrant 전체 컨텍스트 저장 채택**: 단순 역할 라우팅은 에이전트 전문성이 진화할수록 부정확해짐. (question, answer, success, agent_id, project_id, timestamp) 저장. Part 6 Stage 2에서 구현.
