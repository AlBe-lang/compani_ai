## 1. 작업 내용

### 1-1. R-03 수정: `MetricsCollector.flush()` 중복 호출 안전화

**문제**: `flush()` 이후 `_metrics` 리스트를 초기화하지 않아 동일 메트릭이 중복 persist 될 수 있었음.

**수정**: `flush()` 마지막에 `self._metrics.clear()` 추가. `flushed` 카운터 로그 추가. `SQLiteStorage.save()`는 이미 `INSERT OR REPLACE` 시맨틱이므로 데이터 중복 문제는 없지만 불필요한 I/O를 방지.

### 1-2. `KnowledgeGraphPort` Protocol 추가 (`domain/ports.py`)

```python
@runtime_checkable
class KnowledgeGraphPort(Protocol):
    async def store_interaction(self, agent_id, role, question, answer, success, project_id, run_id) -> None: ...
    async def store_task_result(self, result: "TaskResult", run_id: str) -> None: ...
    async def find_best_responder(self, question, context=None) -> str | None: ...
    async def get_expertise_level(self, role, topic) -> float: ...
```

헥사고날 아키텍처 원칙에 따라 도메인 레이어에 프로토콜을 정의하고 구현체는 adapters/application에 격리. 이를 통해 미래에 Qdrant → Pinecone, ChromaDB 등으로 교체 시 이 Protocol 구현만 교체하면 된다 (Rule 10 §3: 교체 가능한 스토리지).

### 1-3. `RedisCache` 어댑터 구현 (`adapters/redis_cache.py`)

- TTL 최솟값 3600s 강제 (04_INFRASTRUCTURE_RULES.md I-08 준수)
- Redis 미실행 시 `connect()`에서 예외 없이 SQLite 폴백으로 자동 전환

### 1-4. `QdrantStorage` 어댑터 구현 (`adapters/qdrant_storage.py`)

- `QdrantClient(path=path)` 로컬 파일 모드 — 별도 서버 불필요
- fastembed (qdrant-client[fastembed] 번들)로 임베딩 — 추가 모델 서버 불필요
- 컬렉션 수동 생성 제거: fastembed의 `add()` 메서드가 첫 삽입 시 named vector config로 자동 생성. 수동으로 `VectorParams`(unnamed) 형태로 생성하면 fastembed의 named vector 검증을 통과하지 못하는 호환성 문제가 있었음 → 자동 생성으로 수정.
- `asyncio.to_thread()` 래핑: QdrantClient는 동기 API → 이벤트 루프 블로킹 방지
- `QARecord` dataclass: Rule 10 §1(컨텍스트 보존) 준수 — agent_id, role, question, answer, success, project_id, run_id, recorded_at 전체 저장
- 컬렉션: `qa_history` (Q&A 이력), `task_results` (태스크 실행 요약)
- 검색 시 컬렉션이 없으면 `[]` 반환 (cold start 안전)

### 1-5. `KnowledgeGraph` 구현 (`application/knowledge_graph.py`)

- **EMA 전문성 추적**: `_expertise: dict[str, dict[str, float]]` 인메모리 테이블. 업데이트: `new = α * sample + (1-α) * prev`, α=0.2, 중립 사전확률 0.5 (Rule 10 §2: EMA 방식).
- **라우팅 우선순위**:
  1. Qdrant 시맨틱 검색 (유사 Q&A ≥ 2개 이상) → 전문성 EMA 최고 역할 선택
  2. 키워드 폴백 — backend/frontend/mlops 3개 키워드 셋
  3. None 반환 (호출부에서 "cto" 폴백)
- `store_interaction()`: Q&A 저장 + EMA 업데이트
- `store_task_result()`: 태스크 결과 저장 + 역할 추론 + EMA 업데이트
- `_detect_topic()` / `_keyword_route()`: 키워드 카운팅 기반 토픽 감지

**주의**: 키워드 매칭은 substring 기반이므로 "ui"가 "build"에 포함되는 등의 오탐이 발생할 수 있음. 현재는 여러 키워드가 명확히 매칭되는 텍스트에서는 정확히 동작하며, 시맨틱 검색이 충분히 누적되면 EMA 기반 라우팅이 이를 보완.

### 1-6. CTOAgent Q&A 루프 구현 (`application/cto_agent.py`)

**문제**: 기존 CTO는 `queue.receive("cto")`를 전혀 호출하지 않아 모든 SLM→CTO 질문이 30초 타임아웃 후 빈 문자열을 반환하는 버그 존재.

**수정 내용**:
- `_QA_PROMPT_PATH` 상수 + `_load_qa_prompt()` 메서드 추가
- `handle_questions(queue)`: `asyncio.create_task()`로 병렬 실행되는 백그라운드 루프. `CancelledError`를 잡아 `cto.qa_loop.stopped` 로그 후 재발생 (클린 종료 보장).
- `_handle_one_question(queue, msg)`: LLM 답변 생성 후 `ANSWER` 타입으로 발신 에이전트 인박스에 전달. 개별 Q&A 오류는 루프를 죽이지 않고 경고 로그만 기록.
- `_generate_qa_answer(msg)`: 실제 LLM 호출 (`max_tokens=512`, `timeout_sec=30`).
- `_build_qa_messages(msg)`: 시스템=qa_response.txt 프롬프트, 유저=질문+컨텍스트(JSON).

**프롬프트 파일**: `application/prompts/cto/qa_response.txt` — ≤300 단어, API 계약·인터페이스 경계 응답 우선, 알 수 없으면 명확히 1문장으로 표명.

### 1-7. `SQLiteMessageQueue` 라우팅 교체 (`adapters/sqlite_message_queue.py`)

- `__init__`에 `knowledge_graph: KnowledgeGraphPort | None` 추가
- `route_question()`: 질문 텍스트를 분석해 최적 에이전트로 라우팅 후 `ask()` 위임
- `_find_responder()`: KnowledgeGraph → 키워드 폴백 → "cto" 순서로 폴백 체인. KnowledgeGraph 예외 발생 시 경고 로그 후 키워드 폴백으로 계속 진행.

### 1-8. `main.py` 완성 — Q&A 백그라운드 태스크 + 신규 인프라 초기화

**`orchestrate_project()` 내부 변경**:
```python
cto_qa_task = asyncio.create_task(cto.handle_questions(queue))
try:
    results = await asyncio.gather(*[_run_task(t) for t in tasks], return_exceptions=True)
finally:
    cto_qa_task.cancel()
    try:
        await cto_qa_task
    except asyncio.CancelledError:
        pass
```
에이전트 실행 완료 후 KnowledgeGraph에 성공 태스크 결과 저장 (`knowledge_graph.store_task_result()`).

**`app_main()` 인프라 초기화 순서**:
1. `QdrantStorage("data/qdrant")` + `await qdrant.init()`
2. `RedisCache("redis://localhost:6379", fallback=storage)` + `await cache.connect()` (Redis 미실행 시 자동 폴백)
3. `KnowledgeGraph(qdrant=qdrant)`
4. `SQLiteMessageQueue(storage, knowledge_graph=knowledge_graph)`
5. Ollama 헬스체크 실패 시 qdrant/cache 모두 close 후 종료
6. 정상 종료 시 `await qdrant.close()` + `await cache.close()` 호출

## 3. 결정 이유

### 왜 `QdrantClient(path=...)` 로컬 파일 모드를 선택했는가?

`docker run qdrant/qdrant` 서버 실행 방식과 비교했을 때:
- 개발·CI 환경에서 Docker 서버 의존성 제거 → 클론 즉시 실행 가능
- `path=":memory:"` 로 테스트에서 I/O 없이 동작
- 전략 문서(04_SYSTEM_ARCHITECTURE.md §4.5)는 Qdrant 사용을 명시하되 배포 방식을 제약하지 않음 → 로컬 파일 모드가 현재 단계에서 가장 단순하고 확장 가능

### 왜 컬렉션 수동 생성을 제거했는가?

`client.create_collection(vectors_config=VectorParams(...))` 는 unnamed vector 형식으로 생성한다. fastembed의 `add()` 메서드는 named vector 형식(`dict`)을 기대하므로 `AssertionError: Collection have incompatible vector params` 가 발생했다. fastembed가 첫 `add()` 호출 시 자동으로 올바른 named vector config로 컬렉션을 생성하므로 수동 생성 코드를 제거했다. `search_qa()` / `search_task_results()`는 이미 컬렉션 부재 시 `[]` 반환으로 cold start를 처리한다.

### 왜 CTOAgent Q&A 루프를 `asyncio.create_task()`로 실행하는가?

에이전트들은 `asyncio.gather()`로 병렬 실행된다. CTO가 동일 이벤트 루프에서 `queue.receive("cto", timeout_sec=1.0)` 를 폴링하려면 독립 태스크로 실행되어야 한다. 별도 스레드는 asyncio Queue의 스레드 안전성을 고려해야 하므로 채택하지 않았다. `finally` 블록의 `task.cancel()` + `await task` 패턴은 에이전트 실행 완료 후 CTO 루프의 클린 종료를 보장한다.

### 전략 변경 이력

전략 문서(04_SYSTEM_ARCHITECTURE.md)에 Redis가 Phase 3+ 인프라로 명시되어 있음을 확인하고 기존 계획대로 Redis 포함 구현 진행. "Qdrant만 사용" 결정이 어느 문서에도 없음을 확인 후 Redis 추가.
