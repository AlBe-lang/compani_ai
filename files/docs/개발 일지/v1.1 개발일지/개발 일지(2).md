### 1-3. 백엔드 변경

**신규 파일**
- `src/interfaces/dashboard_api/runner.py` — `RunManager` (single-flight subprocess
  lifecycle: start / cancel / SSE stream) + `RunState` 데이터클래스. ``main.py
  "<request>"`` 를 `asyncio.create_subprocess_exec` 로 띄우고, stderr 줄 단위로
  버퍼(최대 5000줄)에 적재. SSE 클라이언트는 누적 라인 + tail 새 라인 + `run.done`
  프레임 순으로 받음. cancel 은 SIGTERM → 5s grace → SIGKILL.

**수정 파일**
- `src/interfaces/dashboard_api/routes.py` — 3개 endpoint 추가 (POST /api/run,
  POST /api/cancel, GET /api/run/stream). SSE 는 EventSource 가 헤더를 못
  보내므로 `?token=` 쿼리 인증(이미 `verify_http` 가 fallback 지원).
- `src/interfaces/dashboard_api/app.py` — `DashboardDeps` 에 `run_manager` optional
  필드 추가. `None` 이면 라우트가 503 반환.
- `main.py` — `_run_dashboard_server()` 에서 `RunManager(project_root=Path.cwd())`

## 3. 결정 이유

### 3-1. subprocess 방식 (Q3 핵심)

대안: in-process 호출. 장점은 R-11B 해결과 동시에 SharedWorkspace 단일
인스턴스로 진짜 라이브 EventBus → SSE. 단점은 변경 범위 ~500줄 + main.py
분리 + 라이프사이클 + 테스트. v1.1 마일스톤(빠른 데모 진입점)에 비해 과함.

**subprocess 가 충분한 근거**: stderr 의 structlog JSON 이미 모든 이벤트
포함 (cto.*, slm.*, ws.item.status, ollama.*, queue.qa.*, meeting.*,
peer_review.*). UI 가 그것만 받으면 시각화 가능. R-11B 의 진짜 요구는
"두 프로세스가 같은 객체를 본다" 인데, **stderr 도 사실상 같은 정보**를
반대 방향으로 흐르는 channel — subprocess 가 부모에게 보고하는 형태.

### 3-2. SSE 선택 (vs WebSocket)

WebSocket 은 양방향. 우리는 **단방향(server → client)** 만 필요. SSE 가
표준 EventSource API + 자동 재연결 + HTTP/1.1 호환 + 프록시 친화적. 다만
EventSource 는 헤더를 못 보내서 `?token=` 쿼리 인증으로 절충(`verify_http`
가 이미 query fallback 지원이라 재사용).

### 3-3. React + Vite (vs Flutter 확장)

Flutter (`ui/ceo_dashboard/`) 는 v1.0.0 의 정식 UI 이지만 Flutter SDK 30분
설치 부담 + R-11B 해결 선행 필요. v1.1 은 OLaLA 와 같은 스택(React+Vite)
으로 빠르게 진입하고, v2.0 product UI 결정 시 Flutter 와 다시 비교.

### 3-4. .txt/.md 만 (Q4)

PDF 추출은 `pypdf` 의존성 + 한글 추출 품질 문제. 이미지 OCR 은 한국어
모델 부재. v1.1 데모 가치 대비 비용 큼. v2.0 에서 정식 multimodal 입력 검토.

### 3-5. 일지 디렉토리 명명 (`v1.1 개발일지`)

`PROJECT_PLAN §13` 에 v2.0 / v3.0 만 명시. v1.1 은 PROJECT_PLAN 외 작업
(데모 진입점 추가). Part 0~8 명명을 따라가지 않고 `v1.1` 로 분리하는 것이
정합 — Part 명명은 PROJECT_PLAN 에 있는 작업에만 사용. v2.0 진입 시
`v2.0 개발일지/` 로 다시 분리.

## 4. 발생한 리스크

### 신규 발견

| ID | 항목 | 심각도 | 발현 시점 | 대응 |
|----|------|--------|-----------|-------|
| **D1** | SSE 단일 클라이언트 가정 — 두 브라우저 탭이 동시에 `/api/run/stream` 열면 두 번째는 `state.events` 처음부터 replay 받지만 같은 데이터를 두 번 처리. v1.1 은 1 클라이언트 가정이라 운영상 문제 없으나, 둘 이상 띄우면 first-come-first-served 가 아니라 dual-tail 인 점 명시 필요. | C | 동시 다중 클라이언트 발현 시 | v2.0 에서 broker 패턴(asyncio.Queue per-client) 으로 정식 처리 |
| **D2** | RunManager 가 in-memory 만 — 서버 재시작 시 진행 중이던 자식 프로세스는 고아가 됨. ``runs`` SQLite 테이블 + crash recovery 가 정식 해법이지만 v1.1 범위 초과. | C | 서버 재시작 + 활성 run | v2.0 에서 영속화. 현재는 사용자가 명시적 cancel 또는 자식 프로세스 자연 종료에 의존. |
| **D3** | Vite proxy 가 5173 → 8000 hardcoded — 사용자가 `dashboard_port` 를 변경하면 proxy 도 같이 변경 필요. | C | 비기본 포트 사용 시 | `vite.config.ts` 에 환경변수 (e.g., `VITE_API_PROXY_TARGET`) 도입 후보. v1.1 데모 한정이라 미적용. |
