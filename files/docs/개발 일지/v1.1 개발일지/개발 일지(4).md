## 1. 작업 내용

### 페르소나
10년차 시니어 QA 엔지니어. 본 리뷰는 CompaniAI dev 브랜치 **전체** 에 대한 철저한(very thorough) 코드 리뷰입니다.

### 리뷰 범위
- **src/** 142개 Python 파일 — 도메인계약, 애플리케이션(CTO/SLM/협업), 어댑터(7종), 대시보드(v1.1), 관측성
- **tests/** 77개 파일 — unit 50개, integration, e2e 7개, benchmark
- **web_demo/** Vite + React + TypeScript SPA (v1.1 신규)
- **main.py** 진입점, runner.py (서브프로세스 오케스트레이션)
- 설정, 문서, 빌드파이프라인

### 분석 방식
**8축 관점**:
1. **테스트 가능성** — 모의/통합/E2E 분류, flaky 패턴, fixture 재사용성
2. **버그 가능성 + 엣지 케이스** — off-by-one, null, 동시성, 부분실패, 정리
3. **에러 처리 일관성** — ErrorCode 분류, catch 범위, 재발생
4. **운영성/관측성** — 로그, 메트릭, trace_id 전파, 진단 속도
5. **회복성** — 프로세스 재시작, 트랜잭션, 고아 프로세스, 네트워크 재연결
6. **보안** — 입력검증, SQL injection, secret 누출, timing-safe, subprocess
7. **문서 vs 실제 일치** — docstring, README, 아키텍처 명세 drift
8. **회귀 위험** — v1.0→v1.1 변경, 기본 OFF 기능, hot-reload, pre-commit 우회

## 2. 결과물 (발견 사항)

### 강점 (통과 항목)
1. **계층 분리 우수** — domain/application/adapters/interfaces 명확하게 분리. 도메인 계약(Pydantic)으로 인터페이스 명시.
2. **비동기 패턴 성숙** — asyncio 기반 병렬실행, 동시성 limiter (Part 8 Stage 1) 도입, 메모리 안전장치 내재.
3. **에러 분류 체계** — ErrorCode enum 6개 범주 (E_LLM, E_PARSE, E_DEPS, E_MSG, E_STORAGE, E_SYSTEM), 일관된 catch/재발생.
4. **로깅 기반 관측성** — structlog JSON 출력, 모든 주요 흐름에 단계 로그. component/run_id 바인딩.
5. **구조화된 테스트** — unit 50개, integration, E2E (slow 마커), mock 정교함. test_backend_stage1.py 등 분명한 의도.
6. **보안 의식** — constant-time token 비교 (timing-safe), 입력 빈 문자열 검증, SQL 파라미터 바인딩, secret .env gitignore.
7. **dependency 거버넌스** — aiohttp/aiosqlite/pydantic/structlog 최소한의 의존성. 폐쇄망 설계.

## 3. 결정 이유

### 즉시 Fix 하지 않는 사유
본 세션에서는 리뷰만 수행하고 fix를 하지 않습니다. 이유:

1. **검증 우선**: 각 발견 사항이 진정한 버그인지, 아니면 설계 의도인지 사용자 confirm이 필요.
2. **영향 범위 파악**: Critical 항목들이 다른 컴포넌트와 의존성이 있을 수 있으므로, 고립된 fix는 회귀 야기 가능.
3. **우선순위 결정**: 모든 항목을 동시에 fix하면 큰 커밋. 분기별로 계획하는 것이 적절.
4. **테스트 추가**: 각 fix마다 단위/통합 테스트를 함께 작성해야 하므로, 순차 진행 필요.

### 후속 작업의 우선순위 근거

**Top 3 시급 항목** (다음 단계: v1.1 finalizing):

1. **B2-Critical: orchestrate_project CTO task cancel 예외 처리** (main.py:98–108)
   - **사유**: v1.1 demo 진입점에서 **파이프라인이 갑자기 실패하는 관계**. CTO task cancel 중 CancelledError가 전파되면 전체 orchestration fail. 사용자는 원인을 모름.
   - **Impact**: 높음. 모든 E2E 실행에서 발생 가능성.
   - **Risk**: Low. exception handler 추가만으로 fix 가능.
   - **시간**: ~1시간.

2. **O1-Critical: trace_id per-run vs per-task** (observability/ids.py)
   - **사유**: 현재 trace_id로는 "어느 task에서 에러가 발생했나"를 파악 불가. 대시보드가 error → task mapping을 못 함. 운영성 저하.
   - **Impact**: 중간. 디버깅 시간이 5배 증가할 수 있음.
   - **Risk**: Medium. trace_id 변경은 로그 parsing에 영향 → E2E 테스트 업데이트 필요.
   - **시간**: ~2시간 (+ test update).

3. **R1-Critical: subprocess cancel 후 parent process cleanup** (main.py:169–196, runner.py)
   - **사유**: 사용자가 cancel 하면 subprocess는 SIGTERM 받지만, parent는 5초 grace period 대기. 이 기간에 parent crash → zombie. 장시간 E2E 도중 cancel 시 전체 대시보드 hang 가능.
   - **Impact**: 높음. 운영 환경에서 critical.
   - **Risk**: High. process group 처리는 platform-specific.
   - **시간**: ~3시간 (cross-platform test 포함).
