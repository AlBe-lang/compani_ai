시킬지 prototype 하는 곳)이 필요하다고 합의 → `/Volumes/개발/compani_ai_dev`

### 1-2. dev 디렉토리 생성

`rsync -a` 로 메인 디렉토리 전체를 복사. 단, 내부 `.venv` 의 shebang 이
원본 절대경로를 가리키는 함정 발견 → `pip install` 실행 시 dev 가 아닌

shebang 함정은 메인 메모리에도 기록 (`project_compani_ai.md`).

### 1-5. OLaLA 데모 분석 — 영감

사용자가 참조한 OLaLA Project (`/Volumes/개발/OLaLA-Project`,
`https://github.com/OLaLa-Project/OLaLA-Project.git`) 의 `web/src/pages/demo/
DemoPage.tsx` 873줄 단일 페이지를 정독. 핵심 패턴:

- React + Vite + TypeScript 단일 페이지 SPA
- 카드 그리드 레이아웃 (메트릭 + 모델 + Prompt + Sources)
- **NDJSON 스트리밍** — 백엔드가 한 줄에 한 이벤트 JSON 송신, 프론트가
  `fetch().body.getReader()` + `\n` split 으로 줄 단위 파싱
- 이벤트 타입별 화면 영역 갱신 (`type === "sources"` / `data.response` /
  `data.done`)
- AbortController + Stop 버튼
- 추천 모델 원클릭 (Recommended)

→ 이 패턴을 차용. CompaniAI 의 멀티에이전트 특성에 맞게 변형:
- 단일 stage 대신 **5단계 파이프라인 timeline**
- LLM 토큰 stream 대신 **stderr structlog JSON 이벤트 stream**
- Sources 카드 대신 **에이전트 노드 다이어그램** + **메시지 로그**

### 1-6. 방향 전환 결정 — Q1~Q3

| Q | 결정 | 근거 |
|---|------|------|
| Q1 (기술 스택) | React + Vite + TS — OLaLA 와 동일 | 메인 v2.0 product UI 후보로 진화 가능. Flutter (`ui/ceo_dashboard/`) 는 SDK 30분 + R-11B 선행 부담 |
| Q2 (백엔드 통합) | 메인 `dashboard_api/` 에 `/api/run` + `/api/cancel` + `/api/run/stream`(SSE) 추가 | 메인 이관 정식 경로. dev 보조 서버로 분리하면 v2.0 통합 시 재작업 필요 |
| Q3 (subprocess vs in-process) | **subprocess** — `main.py "<request>"` 자식 프로세스 + stderr SSE 중계 | in-process 는 R-11B 해결 = ~500줄 + 라이프사이클 + 테스트. v1.1 데모 가치 대비 과함. v2.0 별도 마일스톤. **subprocess 도 stderr structlog 가 모든 이벤트 포함하므로 시각화 가치 충분** |

이후 작업은 v1.1 개발 일지(2) 에서 다룬다.

### 1-7. 1차 시도 회고 — 무엇을 배웠나

1. **사용자 의도 명확화 우선**: "데모"라는 단어 하나로 두 가지 다른
   대상(개발자 모니터링 vs end-user UX)을 혼동했다. 다음에는 §9 절차의
   필수 질문 단계에서 "어떤 관점/페르소나의 데모인가" 를 명시 질문에
   포함해야 함.
2. **Streamlit 의 chrome 한계**: Streamlit 의 sidebar/built-in 위젯이 강력
   하지만 그 자체가 "툴 느낌"을 만든다. product UI 가 목표면 React 같은
   자유도 큰 스택이 결국 정합. Streamlit 은 ML 실험·내부 대시보드용.
3. **외부 폴링 vs 내부 통합**: 폴링 도구는 "관찰" 만 가능. 사용자
   인터랙션(입력 → 실행 → 진행 → 결과) 흐름은 백엔드와 직접 통합되어야
   자연스러움. v2.0 R-11B 해결의 동기 일부.
4. **빠른 실험 자체는 가치 있음**: Streamlit 1차 시도가 폐기됐지만 ~30분
   투자로 사용자가 실제로 보고 "아 이게 아니다" 를 확인. **틀린 방향을
   빨리 발견하는 것도 진보**. 매몰비용에 매이지 않는 게 핵심.

## 3. 결정 이유

### 3-1. dev 디렉토리 생성 (vs 기존 메인에서 작업)

메인은 v1.0.0 마감 직후 GitHub 에 push 된 상태. **메인은 v1.0.0 시점의
스냅샷**으로 보존하고, 새 변경은 별도 공간에서 검증 후 cherry-pick 하는
흐름이 안전. dev 가 검증 공간 역할.

### 3-2. Streamlit 1차 시도 (vs 처음부터 React)

당시 시점에는 사용자 의도가 모호("UI 로 실시간 확인하고 싶다"). React +
Vite 는 셋업 비용 + 학습 곡선. Streamlit 은 venv 추가 설치만으로 바로
가능 → "**일단 빠르게 무언가 보여주고 사용자 피드백 받기**" 가 합리적
선택. 결과적으로 폐기됐지만 사용자 의도 발견의 비용으로 30분은 합리적.

### 3-3. 1차 시도 폐기 후 venv 의존성 정리 (vs 그대로 두기)

streamlit 19개 동반 의존성 (~50MB) 이 그대로 있으면 dev 가 점점 어수
선해짐. 깨끗한 격리가 dev = 검증 공간 정신과 부합. 또한 향후 React 작업
중 streamlit 잔재가 헷갈림 유발 가능. 정리 비용은 1분.

### 3-4. OLaLA 패턴 차용 (vs 처음부터 새 디자인)

OLaLA 는 사용자가 직접 협업한 프로젝트, mlops 팀원이 만든 데모를 사용자
가 익숙해함. **사용자 친숙성 + 검증된 패턴** = ROI 최고. 새 디자인은
v2.0 정식 product UI 시점에 다시 평가.

### 3-5. subprocess 방식 (vs in-process, Q3)

R-11B 해결(in-process EventBus 공유)은 v2.0 배포 아키텍처 정비의 일부
이고 ~500줄 변경. v1.1 의 핵심은 "사용자가 데모를 빠르게 본다" 이지
"아키텍처 발전" 이 아님. subprocess 는 stderr structlog 만으로 모든
이벤트(cto.* / slm.* / queue.qa.* / meeting.* / peer_review.*) 를 송신
하므로 시각화 가치 충분. v2.0 에서 in-process 로 자연 진화.
