## 2. 작업 내용

### 2-1. P-12A — 시스템 RAM 카드 동작 (Sprint A)

**원인**: `src/interfaces/dashboard_api/snapshot.py:114 environment_snapshot` 가 `total_memory_gb` / `can_use_e5_large` / `current_embedding_preset` 만 반환. 그러나 `web_demo/src/DemoPage.tsx` 의 메트릭 카드는 `envInfo?.available_memory_gb` 를 표시. **백엔드 응답에 해당 필드 자체가 없어서 frontend 가 항상 fallback `--` 표시.**

**수정**: snapshot.py 에 두 필드 추가
```python
vm = psutil.virtual_memory()
total_gb = vm.total / (1024**3)
available_gb = vm.available / (1024**3)
used_pct = float(vm.percent)
```

**검증**: `curl /api/environment` 응답에 `available_memory_gb=5.93`, `memory_used_pct=62.9` 가 정상 출력됨을 확인.

**부수 발견**: macOS 의 `vm.available` 은 free 가 아닌 `free + inactive` 의 의미라 표시되는 5–6 GB 가 자연스러움. vm_stat 의 Pages free 만 보면 ~70 MB 로 매우 작음 (시스템 page cache 때문).

**모달 디자인**:

### 2-4. P-12D — 진짜 모드의 hang 위험 회피 (Mock 토글)

**증상 재현**:
- 사용자가 `./demo` 실 모드로 "블로그" 요청 후 한참 멈춤
- 두 번째 시도로 "Todo" 입력 → 백엔드 RunManager 가 `is_running()=False` 로 판단 (이전 finished.set 됐거나 race) 하여 두 번째 subprocess 시작
- `ps aux | grep main.py`: pid 89826 (블로그) + pid 94167 (Todo) **둘 다 살아있음**
- `ollama ps`: 비어있음 — Ollama 가 요청을 받았지만 16GB Mac swap 으로 응답 없음
- 19 분 동안 어떤 stderr 도 안 들어옴 → SSE 스트림 정지

**원인**: Part 8 시니어 QA 리뷰의 R1-Critical (subprocess cleanup 미흡) 의 실제 재현. 추가로 R-11A (peer_review_mode 등 일부 분류 재조정) 와 별개로, **두 subprocess 동시 살아있을 때 메모리 압박이 두 배** 가 되어 ollama 자체가 응답 멈춤.

**해결 — 토글 + confirm 안전장치 (사용자 결정으로)**:
세 가지 옵션을 검토:
- A. 토글로 mock/real 전환 (한 디렉토리)
- B. mock 디렉토리 별도 분리 (두 시스템)
- C. real 모드 그대로 유지 + warm-up 추가

선택: **A + 안전장치** — 한 디렉토리에서 토글로 둘 다 시연 가능. 잘못된 클릭 방지를 위해 다음 안전장치 추가:

1. **명시적 라벨**: 🟢 `MOCK · 시연 모드 (38초)` ↔ 🔴 `REAL · 실제 LLM (5-10분)` — 색깔로 즉시 구분
2. **REAL 진입 시 confirm 모달**: 시간 / 위험 / 메모리 권고 4 항목 명시 + "취소 (Mock 모드 유지)" 가 default 강조
3. **백엔드 분기**: `RunStartRequest.mock: bool = False` (기본 안전한 쪽), `RunManager.start(mock=True)` 시 `scripts/mock_pipeline.py` spawn

**시연 본 트랙**: 토글 ON 으로 시작 → 38 초 시연 100% 안전. 교수 즉석 요청 시 토글 OFF + confirm → 양해 멘트 자연스러움.

### 2-5. 산출물 자동 다운로드 + 미리보기 (Sprint D)

**기존**: 파이프라인 종료 후 산출물이 `outputs/<slug>/` 에 저장되지만 사용자는 터미널로 `open outputs/...` 해야 확인 가능.

**추가**:
- **3 개 backend endpoint** (`src/interfaces/dashboard_api/routes.py`):
  - `GET /api/output/download?slug=<slug>` — `outputs/<slug>/` 를 `zipfile.ZipFile(ZIP_DEFLATED)` 로 압축 + `Content-Disposition: attachment` 응답
  - `GET /api/output/tree?slug=<slug>` — 디렉토리 트리 JSON (재귀 `_build_tree`)
  - `GET /api/output/file?slug=<slug>&path=<path>` — 텍스트 파일 미리보기 (1 MB 한계, path-traversal 방어)
- **Frontend**: `orchestrator.done` 이벤트 시 `<a download>` 트릭으로 자동 ZIP 다운로드 트리거 + 결과 카드에 "📂 산출물 미리보기" 버튼 → 좌측 트리 + 우측 코드 미리보기 패널 (`OutputTreePanel` + `TreeNode` 컴포넌트)

**검증 결과**:
- ZIP 크기 10,844 bytes, 15 파일 (FastAPI 백엔드 + React 프론트엔드 + Docker + GitHub Actions CI) 모두 포함
- 발표 시연 흐름: 종료 → 자동 다운로드 → "산출물 미리보기" 클릭 → `backend/main.py` 가 청중에게 보임

## 3. 결정 이유

### 3-1. 왜 토글 방식 (한 디렉토리) 인가 — B 안 (디렉토리 분리) 거절

(B) mock 디렉토리를 완전 별도 프로젝트로 만드는 안도 검토. 장점은 시연 안정성 — 두 시스템이 절대 섞이지 않음. 그러나 다음 단점이 더 큼:
- **교수 즉석 요청 — "실제로도 돌려볼 수 있어요?"** — B 안에서는 다른 디렉토리로 전환해야 해서 어색
- **코드 중복** — frontend 와 백엔드 일부를 두 벌 유지해야 함
- **청중에게 두 시스템 설명 필요** — 시연 컨텍스트 단순화 깨짐

(A) 토글 + confirm 의 잘못된 클릭 위험은 안전장치 3 종으로 거의 0 — 색깔 / 라벨 / confirm 모달 + default 취소.

### 3-2. 왜 풍부한 payload 를 mock 에 만 추가했나 — backend orchestrator 도 같이 수정해야 하지 않나

수정해야 함. 다만 **v1.1 발표 우선순위** 로 mock 만 우선 풍부화. 실 backend orchestrator 의 이벤트 schema 통일은 v1.2 작업으로 이월 (Stage 4-A — see §4 미해결).

### 3-3. 왜 산출물 자동 다운로드인가 — 사용자가 직접 클릭하게 두는 게 더 자연스럽지 않나

청중 입장에서 "끝났습니다 → 자동 다운로드 → 클릭" 이 한 흐름으로 이어지는 게 시연 몰입도 ↑. 단, 다운로드 차단된 브라우저 환경 대비 **수동 다운로드 버튼** 도 함께 제공 (`📂 산출물 미리보기` 옆 `⬇️ 다시 다운로드 (ZIP)` 버튼).

### 3-4. 왜 1 Hz tick 으로 강제 리렌더 하나

진행중 단계의 경과시간을 표시하려면 매초 갱신 필요. 그런데 stage state 자체는 변하지 않으니 React 가 리렌더하지 않음. `useState<number>(0)` 에 `setInterval` 1 초마다 increment → 강제 리렌더. 단순하지만 효과적. 단, 모든 state 가 다시 평가되어 약간의 오버헤드 — 발표용으로는 무시 가능 수준.
