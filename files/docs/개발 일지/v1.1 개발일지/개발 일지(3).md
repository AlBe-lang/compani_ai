# v1.1 개발 일지 (3) — 8축 종합 코드 리뷰 (push 직전)

> 작성일: 2026-04-27 | 단계: v1.1 검증 | 작업 디렉토리: `/Volumes/개발/compani_ai_dev`

## 1. 작업 내용

dev3 브랜치 push 직전, 본 세션에서 추가/수정한 코드(메인 dashboard_api 확장
+ web_demo SPA + 일지)에 대해 **8축 종합 코드 리뷰**를 수행했다. 사용자가
요청한 "코드 리뷰" 의 정의는 명시되지 않았으므로, 일반적 ultra-thorough
multi-axis review 로 해석하여 다음 8축에 대해 정적 분석:

1. 아키텍처 (의존성 방향, 계층 분리, Ports & Adapters)
2. 코드 품질 (가독성, 네이밍, 중복, dead code)
3. 테스트 (커버리지, 단위/통합 분리, mock 사용, edge case)
4. 보안 (토큰, injection, secret 노출, 인증 우회)
5. 성능·메모리 (동시성, 백프레셔, 버퍼 cap, re-render)
6. 운영성 (로깅, ErrorCode, recovery, 고아 프로세스)
7. 문서·컨벤션 (일지 4섹션, COMMON_RULES.md 준수, 커밋 메시지)
8. 메인 이관 정합성 (v1.0.0 패턴과 충돌 없는지)

리뷰 범위: 본 세션에서 변경된 파일만 (총 ~800 LOC backend + ~770 LOC
frontend + 일지).

### 🟠 Major (3)

#### M1 — React re-render 폭주 가능성
- **위치**: `web_demo/src/DemoPage.tsx:160 handleLogEvent`
- **현상**: SSE 이벤트 수신마다 `setEvents(prev => ...)`, `patchStage`, `patchAgent`
  를 동기 호출 → 빠른 stream(예: 초당 50줄)에서 React reconciliation 매 라인.
- **영향**: 데모 시 UI 끊김, CPU 점유 상승.
- **권장**: `requestAnimationFrame` 또는 100ms `setTimeout` batch update.
- **분리**: v1.2 후보 (10줄 변경).

#### M2 — 자식 프로세스 고아화
- **위치**: `src/interfaces/dashboard_api/runner.py` + `main.py:_run_dashboard_server`
- **현상**: dashboard 프로세스가 비정상 종료 시 `RunManager` 메모리만 사라지고
  자식 `main.py "<request>"` 는 계속 살아남음.
- **영향**: 토큰/리소스 낭비, 다음 dashboard 재시작 시 PID 추적 불가.
- **권장**: FastAPI `@app.on_event("shutdown")` → `run_manager.cancel()`.
- **분리**: v1.2 후보 (5줄 변경). 일지(2) §4 D2 와 동일.

#### M3 — subprocess 라이프사이클 통합 테스트 부재
- **위치**: `tests/unit/test_dashboard_runner.py`
- **현상**: SIGTERM → 5초 grace → SIGKILL 흐름은 mock 으로만 검증, 실제
  시그널 핸들링은 통합 테스트 영역.
- **영향**: cancel 이 진짜 멈추는지 시각 확인만 가능.
- **권장**: `tests/integration/test_runner_subprocess.py` — `python -c "import
  time; time.sleep(30)"` 더미로 SIGTERM/SIGKILL 검증.
- **분리**: v1.2 또는 v2.0 후보.

## 3. 결정 이유

### 3-1. Major 3건 즉시 fix 안 하고 push 진행

**이유**:
- Critical 0건이라 push 차단 사유 없음.
- M1 (re-render): UI 미세 끊김이지 차단 버그 아님. 데모 가치는 보존.
- M2 (고아 프로세스): 비정상 종료 시에만 발현. 정상 종료 흐름은 cancel 자동 호출.
- M3 (통합 테스트): 단위 테스트로 라우트/state 분기는 충분히 커버됨.
- **본 세션 범위는 v1.1 데모 진입점**. M1~M3 는 모두 v1.2 또는 v2.0 자연
  발전 항목 — 한 번에 끼워넣으면 PR 추적 어려움. 일지화 후 별도 작업 권장.

### 3-2. 8축 균형 리뷰 (vs 특정 축 강조)

데모 entry point 는 사용자 입력 → LLM 호출 → 파일 저장까지 모든 layer 를
관통. 어느 한 축만 깊게 보면 다른 축의 위반 놓침. 8축 균형이 정합.

### 3-3. 즉시 fix 후보 (선택)

push 전에 fix 가능한 작은 항목:
- **S1** (50KB cap): 5줄, 즉시 가능. 그러나 데모 시 사용자가 큰 파일 안
  올릴 가능성 높음 → 일지 기록만으로 충분.
- **D1** (web_demo/README): 30줄, 즉시 가능. 일지(2) §3-5 의 라이브 검증
  가이드와 중복 → 일지로 충분, README 는 v1.2 후보.
- **C2** (`Depends(_auth)` 일관성): 2줄. SSE query token 의도 docstring 으로
  보존 가능 → 일관성 vs 의도 명시 trade-off, prototype 단계는 의도 명시 우선.

→ **즉시 fix 0건, 모두 일지화 후 push** 결정.

### v1.0.0 잔존 (영향 받음)

- **N3** (SLM 90s timeout RED) — 데모 실행 시 Backend Agent fail 가능. 별도
  PR 로 SLM timeout 상수 상향 + ErrorCode 분류 보강. **v1.1 다음 작업 1순위**.

### 본 일지 자체의 한계

- 정적 분석 only. **동적 검증** (실제 띄워서 동작 확인) 은 사용자 환경에서
  진행. 일지(2) §3-5 라이브 검증 가이드 참조.
- **보안 침투 테스트** 미실시 — 정적으로 토큰/injection 만 점검. 실제 공격
  시뮬레이션은 별도 도구 필요 (production 진입 시).
- **성능 측정값** 없음. M1 의 re-render 폭주는 추정 — 실제 stream 빈도 측정
  후 우선순위 재평가 필요.
