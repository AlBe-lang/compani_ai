### 1-4. R-06A — pre-commit 훅 도입

**신규 파일**:
- `.pre-commit-config.yaml` — black/isort/flake8 (격리 venv) + mypy (local hook, project venv 자동 분기)

**mypy hook 처리**:
mirrors-mypy 의 격리 venv 는 src 의 런타임 의존성(pydantic 2.13, fastapi 0.136 등)을 모르기 때문에 false-positive 다수 발생. 이를 회피하기 위해 `language: system` + bash 분기로 처리:

```yaml
- repo: local
  hooks:
    - id: mypy
      name: mypy (project venv)
      entry: bash -c 'PY=python; [ -x .venv/bin/python ] && PY=.venv/bin/python; "$PY" -m mypy "$@"' --
      language: system
      types: [python]
      pass_filenames: false
      args: [src, tests, main.py]
```

로컬은 `.venv/bin/python`, CI 는 시스템 `python` 자동 사용.

**수정 파일**:
- `requirements-dev.txt` — `pre-commit>=3.0.0,<5.0.0` 추가
- `Makefile` — `pre-commit-install` 타겟 추가 + `.PHONY` 갱신
- `.github/workflows/ci.yml` — lint job 의 직접 black/isort/flake8/mypy 호출을 `python -m pre_commit run --all-files --show-diff-on-failure` 단일 호출로 교체 (로컬·CI hooks 동일성 보장)

**사전 검증**: `pre-commit run --all-files` 4종 hook 모두 통과(N3 부재 확인).

### 1-5. R-11A — HOT_RELOADABLE 분류 재조정

**수정 파일**:
- `src/interfaces/dashboard_api/config_mutation.py` — `HOT_RELOADABLE` frozenset 에서 8개 필드 제거:

  - `rework_enabled`, `rework_max_attempts`

이들은 자동으로 `RESTART_REQUIRED` 로 분류된다(`classify()` fallback). 진짜 hot reload 가능한 것은 `LLMConcurrencyLimiter.update_limits()` 경유 `llm_concurrency_*` 4종뿐.

주석도 갱신해 "Peer review / rework — components now read via SystemConfig ref" 거짓 설명을 R-11A 강등 사유로 교체.

**테스트 갱신** (`tests/unit/test_dashboard_routes.py`):
- `test_get_config_returns_per_field_metadata` — `peer_review_mode` 의 category 검증을 `restart_required` 로 변경. hot_reloadable 예시는 `llm_concurrency_slm` 으로 교체.
- `test_patch_config_hot_reloadable_succeeds_without_confirm` — `llm_concurrency_slm` 으로 교체 + limiter propagation 검증.
- `test_patch_config_restart_required_succeeds_without_confirm` — R-11A 강등된 필드도 destructive 가 아니라 PATCH 자체는 confirm 없이 통과해야 함을 검증하는 신규 테스트 추가.

## 3. 결정 이유

### 3-1. R-06A 에 mypy 를 local hook 으로 둔 이유

`mirrors-mypy` 의 표준 hook 은 격리 venv 에서 실행되므로 src 의 런타임 의존성을 모름. 모든 의존성을 `additional_dependencies` 에 명시하면 requirements.txt 와 이중 관리가 되어 drift 발생 위험. 본 프로젝트는 venv 를 표준 위치(`.venv/`)에 두는 컨벤션이 정착되어 있으므로(`Makefile` `PYTHON ?= python`), `language: system` + bash 자동 분기가 가장 단순·안전한 해법.

### 3-2. R-11A 에서 (a) 분류 재조정만 적용하고 (b) ref 매번 재조회 미적용

(b) 는 PeerReviewCoordinator/ReviewerSelector/ReworkScheduler/EmergencyMeeting 4개 컴포넌트의 `__init__` + 메서드 다수를 손대야 하며, 각 컴포넌트가 보유한 dataclass 사본(`PeerReviewConfig` 등)도 함께 변경 필요. 변경 면적이 크고 회귀 위험 높음. (a) 는 frozenset 5줄 + 테스트 갱신만으로 사용자 혼란(UI 배지 거짓) 직접 제거. ROI 우월.

(b) 의 진짜 hot reload 욕구는 v2.0 멀티 프로젝트 동시 진행 단계에서 다시 평가하는 것이 자연스러움 (그때는 SystemConfig 자체가 더 큰 재설계 대상).

### 3-3. CI lint job 을 pre-commit 호출로 교체

기존 CI lint job 은 `black --check` / `isort --check-only` / `flake8` / `mypy` 직접 호출. 새로 도입한 `.pre-commit-config.yaml` 과 동등한 검사. 두 곳 동시 유지는 drift 위험(예: 한쪽에서 black 24.12 로 올렸을 때 다른 쪽 누락). 단일 호출(`pre-commit run --all-files`) 로 통합해 SSOT 확보.

### 3-4. Q3 BREAKING CHANGE 태그 미적용

`HOT_RELOADABLE` frozenset 변경은 `/api/config` 응답의 `category` 필드 *값* 을 바꾸지만 SystemConfig 의 *필드 자체* 는 동일. domain/contracts 스키마는 불변. §2.5 BREAKING CHANGE 가 보호하는 대상은 도메인 스키마이므로 적용 대상 아님. 다만 일지에는 변경 사실 명시(독자 추적 가능).

### 3-5. Q5 git tag v1.0.0

PROJECT_PLAN §8 "Phase 8 시스템 완성" 도달 표식. CHANGELOG/RELEASE 별도 문서는 다음 작업(v2.0 첫 PR) 과 함께 신설하는 것이 자연스러움 — v1.0 가 완성품이 아니라 마일스톤이며, 첫 외부 release 는 v2.0 의 GitHub 연동·HITL 도입 시 의미를 가짐.
