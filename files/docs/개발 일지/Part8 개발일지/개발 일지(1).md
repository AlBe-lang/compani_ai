### 1-4. 핵심 로직 요약

**LLMConcurrencyLimiter**
```
per-role semaphore + total semaphore (double-gate)
acquire 순서: total → per-role → yield
unknown role → 비제어 yield (defensive)

기본값 (Mac Mini 16GB):
  cto=1, slm=1, mlops=1, total=2
  → 동시 2개 모델까지 (CTO + 1 SLM 또는 2개 다른 SLM 역할)
참조: 04_INFRASTRUCTURE_RULES.md I-04, 04_SYSTEM_ARCHITECTURE.md §7.3
```

**EmbeddingPreset + QdrantStorage 자동 migration**
```
SystemConfig.embedding_preset = MPNET_BALANCED (기본)
  → QdrantStorage(embedding_model="sentence-transformers/...mpnet-base-v2")
  → vector_size=768 자동 계산
  → init() 시 컬렉션 존재하면 차원 비교:
       일치: 유지
       불일치 + allow_recreate=True: WARN + delete + create
       불일치 + allow_recreate=False: RuntimeError (명확한 설명)
```

**OllamaProvider timeout_sec**
```
raw_timeout = kwargs.get("timeout_sec", 300)
timeout_sec = min(float(raw_timeout), _MAX_TIMEOUT_SEC=300)  # I-02 하드캡
aiohttp.ClientTimeout(total=timeout_sec)
```

**벤치마크 이원화**
```
make test         → -m "not slow and not benchmark" (벤치 제외)
make bench-mock   → pytest -m benchmark (회귀 감지, 상대치만)
make bench-real   → python scripts/benchmark_real.py (실 Ollama 절대치)

리포트 mode 명시:
  mock 리포트: [bench mock] 헤더
  real 리포트: JSON 에 "mode": "real" + warning 필드
  → 두 숫자 혼용 방지
```

## 3. 결정 이유

### 3-1. 왜 벤치마크 이원화에 Make 타겟 분리가 핵심인가 (Q1 심화)
사용자 우려는 "두 숫자가 충돌하면 어느 쪽 믿지?" — 이 문제는 **물리적 분리** 로 해결됨:
- 서로 다른 명령 (`bench-mock` vs `bench-real`) 실행
- 서로 다른 경로 (`benchmarks/reports/mock/` vs `benchmarks/reports/real/`) 출력
- 리포트 헤더에 mode 명시
- 사용자는 의도적으로만 섞을 수 있음 — 사고로 섞일 수 없음

이 설계가 없으면 (C) 의 단점이 장점을 상쇄. 설계가 있으면 장점이 단점을 능가.

### 3-2. 왜 응답 캐싱을 전면 skip 하는가 (Q2 심화)
- 실제 cache hit 기회 분석 결과 "단일 run 내 0회, cross-run 에서도 동일 prompt 재현 희박"
- 애플리케이션 레벨 캐싱 = overhead(hash, Redis roundtrip ~1-2ms) 만 추가
- Ollama 자체 KV cache 가 이미 동일 system prompt 연속 호출 시 활용
- **절약된 개발 시간을 Q5 병렬화에 투자** → 체감 효과 훨씬 큼

향후 운영 중 "동일 prompt 반복" 패턴이 실제 발견되면 그때 narrow 도입 검토 (Phase 9 또는 v2.0).

### 3-3. 왜 mpnet 을 바로 채택했나 (Q3 심화)
- Mac Mini 16GB 메모리 실측: CTO 8B + SLM 4.5B + 시스템 = 15.8GB, 여유 0.2GB 타이트. MiniLM 117MB + mpnet +300MB = 0.1GB 여유 → **fits, but barely**
- MTEB 다국어 벤치 공개 수치: mpnet 이 MiniLM 대비 +15% — 재측정 불필요
- 벤치마크 반나절 대신 **직접 도입 + rollback 조건 명시** 가 시간 경제적
- Rollback 1줄 변경 (`EmbeddingPreset.MINILM_FAST`) — on/off 설계로 비용 0

### 3-4. 왜 on/off 로 EmbeddingPreset 을 지원했나
사용자 Q3 재질문: "Q1 처럼 on/off 안 되나?" — enum 기반 `EmbeddingPreset` + `QdrantStorage` 차원 자동 reconciliation 이 실질적 on/off. 환경변수 / CLI 플래그 / CEO 대시보드 (Stage 2) 에서 조정 가능. 차원 불일치 시 컬렉션 자동 재생성으로 R-05C 요구사항 자연 통합.

### 3-5. 왜 EmbeddingProviderPort 추상화를 이월하는가 (Q4 심화)
- Q3 결정 (mpnet 교체) 은 Port 없이도 1줄 변경으로 완료 — abstraction 불필요
- Rule 10 §3 는 이미 `KnowledgeGraphPort` + `StoragePort` 로 충족 (벡터 DB 교체 자유)
- 임베딩 공급자 교체 시나리오 (OpenAI 등) 는 imminent 아님 — **YAGNI**
- Stage 1 범위 팽창 방지 (Rule 9.1 P-04) — 이미 5개 주제 다룸

Stage 3 "잔존 리스크 마감" 에서 "도입 / 유보 / 기각" 3택 중 결정 예정.

### 3-6. 왜 모델별 상한 dict 인가 (Q5 심화)
- `04_INFRASTRUCTURE_RULES.md` I-04 철학 ("큰 모델 동시 금지") 은 모델 크기 기반 제약 — dict 가 자연
- Mac Mini 16GB 실측: CTO 8B + SLM 4.5B = 14.3GB + OS 1.5GB = 15.8GB **바로 swap 임계**
- 2개 SLM 동시 = 19.2GB → 확정 swap → 성능 체감 30~50% 악화
- 모델별 + 전체 이중 상한으로 **어느 차원도 절대 초과 안 함**

### 3-7. 왜 main.py 의 기존 `asyncio.gather` 를 그대로 두고 semaphore wrapper 만 추가했나
main.py:87 가 이미 gather 사용 중 — 즉 **이미 병렬 실행 구조**. 문제는 **무제한 동시성** 이었음. 구조 재작성 대신 `async with limiter.limit(role):` 한 줄 wrapping 으로 해결. 최소 변경 원칙.

### 3-8. 왜 R-05A (fastembed mean-pooling warning) 는 별도 코드 변경 없이 처리했나
- Warning 자체는 informational (functional 영향 0)
- `add_custom_model` 으로 CLS pooling 복원은 **MiniLM 유지 시에만** 의미
- Q3 에서 mpnet-base-v2 로 교체 확정 — mpnet 은 mean pooling 이 표준 (sentence-transformers 기본)
- 결론: warning 무시, 코드 변경 없음. 일지에 결정 기록.

### 3-9. 왜 KG load_expertise 를 main.py 에 추가했나
Part 7 Stage 3 에서 KG `_expertise` SQLite 영속화를 도입했으나 main.py 가 이를 활용 안 함. Stage 1 정리 중 발견 — 시작 시 `load_expertise()` 호출 안 하면 영속화 효과 0. 추가: `KnowledgeGraph(qdrant, storage=storage)` + `await knowledge_graph.load_expertise()`.
