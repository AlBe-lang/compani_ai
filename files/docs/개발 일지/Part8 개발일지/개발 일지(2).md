## 3. 결정 이유

### 3-1. 왜 디렉토리 구조 (Q1)

hexagonal `src/interfaces/` 계층 유지. CLI 대시보드(`cli_dashboard.py`) 와 같은 레이어에 웹 대시보드를 두면 "interface = 사용자 접점" 의미 일관성이 명확. `services/` top-level 이동은 **필요해지면 30초 `git mv`** 로 가능 — YAGNI 원칙 (Rule 9.1 P-04).

### 3-2. 왜 초기 snapshot 자동 push 추가 (Q2 보강)

(C) 하이브리드 원안은 이벤트 + 주기 snapshot 이지만 **첫 접속 순간엔 아무 데이터도 표시 안 됨** (이벤트가 아직 없고 첫 tick 은 5초 뒤). 이를 해결하기 위해 **접속 직후 서버가 현재 상태 전체를 snapshot 한 번 발송**. 사용자 체감 0초 부팅.

snapshot 내용은 SharedWorkspace `_cache` + DNAManager `_cache` + MetricsCollector 즉시 조회 — 별도 스냅샷 저장소 불필요.

### 3-3. 왜 3-category 분류 (Q3 D) — 그리고 실제 hot-reload 가능 필드는 어디까지 (정직한 기술)

분류 메타데이터는 `config_mutation.HOT_RELOADABLE` / `DESTRUCTIVE` frozenset. 나머지는 자동으로 `RESTART_REQUIRED`.

**실제 구현 수준**:
- `llm_concurrency_*` — `LLMConcurrencyLimiter.update_limits()` 호출로 세마포어 교체됨. **실제 즉시 반영**.
- `peer_review_mode`, `rework_enabled` 등 — 현재 components 가 init 시 `PeerReviewConfig`/`ReworkConfig` 객체로 값을 **복사** 해서 들고 있음. SystemConfig 를 mutate 해도 즉시 반영 안 됨.
  - Stage 2 에선 UI 가 "즉시 적용" 이라고 표시하지만 실 효과는 "즉시 + 다음 컴포넌트 재초기화 시" 정도.
  - 진정한 hot-reload 는 컴포넌트 리팩토링 필요 (component 가 SystemConfig ref 를 보유하고 매번 재조회) — Stage 3 검토.

**정직한 문서화**: 일지 §4 R-11 시리즈에 이 한계 명시. UI 사용자에게도 "즉시 적용" 배지가 현재는 일부 필드만 실효 있음을 RUNBOOK 에 추가 기술 예정.

### 3-4. 왜 Flutter SDK 사용자 직접 설치 (Q4 모노레포)

실무 표준 패턴 (Serverpod, Shorebird 등). 개발자 환경 사전 요구사항 = Python venv 설치와 동일 원리. **최종 사용자는 Flutter 불필요** — FastAPI 가 빌드된 정적 자산 서빙.

`build/` 은 .gitignore — CI 에서 빌드하거나 개발자가 로컬 `make flutter-build` 실행 후 uvicorn 이 같이 서빙.

### 3-5. 왜 Jupyter-style 자동 토큰 (Q5)

- `127.0.0.1` 바인드만으로도 외부 공격 차단되지만 **같은 맥북의 다른 프로세스** 는 접근 가능
- Jupyter/Ollama Web UI 가 정확히 이 패턴 — 검증된 UX
- 토큰은 매 run 마다 재생성 (`SystemConfig.dashboard_token=""` 기본값이 falsy → uuid4)
- stderr 에 배너 (Jupyter 모방) — stdout 파이프라인 오염 없음

### 3-6. 왜 WebSocket 통합 테스트를 broadcaster 단위로 대체했나

`TestClient.websocket_connect` 는 내부 anyio portal 로 async→sync 브릿지. 외부에서 `asyncio.run(publish)` 은 다른 이벤트 루프 → `broadcaster._on_event` 의 `get_running_loop().create_task` 가 해당 루프에 태스크 예약 → publish 완료와 함께 루프 종료 → 태스크 취소됨. 즉 **테스트 인프라 제약**이지 코드 버그 아님.

대체: `DashboardBroadcaster` 를 직접 호출하는 2개 단위 테스트로 fan-out 로직 검증. 실 production 흐름 (uvicorn 안에서 모든 await 이 한 루프에 있음) 에서는 문제 없음.

### 3-7. 왜 `sample_memory` 를 추가했나 (R-10A)

Stage 1 에서 mpnet 임베딩이 Mac Mini 16GB 여유 0.8GB 타이트하다는 리스크. 실 운영 중 **실제로 swap 직전까지 갔는지 관측**할 방법이 없으면 문제를 감지 못함. `psutil.Process().memory_info().rss` 기반 peak 추적은 거의 무비용(마이크로초) + run별 누적.

`/api/metrics` + `metrics_tick` 이 `memory_peak_gb` 실시간 노출 → UI 홈 페이지 카드로 표시 → 사용자가 설정 변경 판단의 근거로 활용.

### 3-8. 왜 E5_BEST 환경 가드 (R-10F)

`/api/environment` 에 `can_use_e5_large: bool` 반환. UI `/settings` 가 메모리 < 24GB 감지하면 배너로 경고 + E5_BEST 옵션 회색 처리. 이는 "자유를 뺏지 않되 발등 찍기 방지" — 사용자가 강제로 E5 선택해도 confirm 모달 거치면 허용.

### 3-9. 왜 `--dashboard` 플래그를 별도 프로세스로 분리했나

파이프라인 실행 (`main.py "<request>"`) 과 대시보드 서버 (`main.py --dashboard`) 를 **한 프로세스에서 동시** 실행하려면 orchestrator + uvicorn 을 같은 이벤트 루프에 통합해야 함. 이는:
- `orchestrate_project` 이 blocking await 체인 → 서버가 그 동안 응답 못 함
- 또는 asyncio.gather 로 묶지만 예외 전파·종료 협상 복잡

Stage 2 는 **별도 터미널 권장**: `terminal A: python main.py --dashboard` + `terminal B: python main.py "todo app"`. 공유 SQLite 파일(`data/compani.db`) 을 통해 대시보드가 진행 상황을 읽음. Stage 3 에서 단일-프로세스 통합 검토.
