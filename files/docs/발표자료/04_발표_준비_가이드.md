## 0. 발표의 본질 — 무엇을 얻으려는가?

이 발표는 **시스템의 시연 + 연구 질문의 학술적 위치를 설명** 하는 자리입니다.

### 청중이 평가하는 것
| 축 | 평가 포인트 |
|----|-----------|
| 문제 의식 | 학술적 의미 있는 RQ 인가? lab 관심 영역과 fit? |
| 기술 깊이 | 시스템 설계/구현 결정의 근거가 명확한가? |
| 연구 설계 | 실험 매트릭스가 RQ 를 검증할 수 있게 짜였는가? |
| 학술 위치 | 기존 연구 (ChatDev / ExpeL / ProtocolBench) 와의 차별점이 명확한가? |
| 한계 인식 | 자원/방법론 한계를 인지하고 보완 계획이 있는가? |

### 발표 후 얻고자 하는 것

## 3. 발표 중 — 핵심 원칙

### 3.1 페이스 컨트롤
- **데모 중**: 슬로우 — 청중이 화면 변화를 따라잡을 시간 필요. 메시지 모달 띄우면 **2-3초 침묵** 후 설명
- **슬라이드**: 한 슬라이드당 평균 1분. 짧으면 30초, 길면 2분 이내
- **Q&A**: 질문에 즉답 X. **"좋은 질문입니다"** 같은 클리셰도 절제. 그냥 잠깐 호흡 후 답변

### 3.2 시선 / 자세
- 슬라이드를 보지 말고 **청중을 보세요**. 슬라이드는 손가락으로만 가리킴
- 데모 중: 마우스 커서로 가리키고 + 청중을 향해 설명
- 손짓: 자연스럽게. 주머니에 손 넣지 말 것
- 자세: 한 자리에 못박히지 말고 슬라이드 → 청중 → 화면 자연스럽게

### 3.3 단어 선택 (정직성 ≥ 화려함)
- "당연히" 사용 금지 — 청중이 모를 수 있음
- "쉽게 말씀드리면" 권장 — 풀어 설명
- 기술 용어는 첫 등장 시 한 번 풀어서 설명 (예: "asyncio — 파이썬 비동기 라이브러리입니다")
- 모르면 **"솔직히 그 부분은 깊이 못 봤습니다. 추정으로는 [X] 같은데, 정확하게는 [Y] 를 더 봐야 합니다."**

### 3.4 데모 시 비상 대응

#### 데모가 멈춘 경우 (mock 인데도 끊긴 경우)
```
1. 너무 흔들리지 말 것
2. "잠시 새로고침 한 번 하겠습니다" → F5
3. 토큰 URL 손에 적어둔 것으로 다시 접속
4. 그래도 안 되면 → 슬라이드 22 의 "비상 화면" 캡처로 대체
   "지금 라이브가 어렵지만, 이게 평소 동작 결과입니다"
```

#### 진짜 모드를 보여달라는 요청
```
거절 멘트 (정직):
"교수님, 진짜 LLM 호출 모드는 16GB Mac 환경에서 5-10분 걸리고
swap 발생 가능성이 있어서, 발표 중 완주를 보장하기 어렵습니다.
대신 이전에 진짜로 돌렸을 때 산출물을 보여드릴 수 있고,
또는 발표 후 별도 시간에 함께 봐주실 수 있을까요?"
```

#### 답을 모르는 질문
```
권장 답변 패턴:
"정말 좋은 지적입니다. 솔직히 그 부분은 [X 정도까지만] 파봤는데,
직관적으로는 [Y] 라고 생각합니다. 다만 정확한 답을 드리려면
[관련 논문 / 추가 실험] 을 더 봐야 합니다.
대학원에서 같이 풀어보고 싶은 부분이기도 합니다."
```

**절대 금지**: 짐작으로 답하기 / 화려한 어휘로 회피 / "좀 어렵네요" 만 하고 끝내기

## 4. Q&A 전략

### 4.1 자주 나올 질문 카테고리 (시연 대본 01 의 16개 질문 참조)

| 카테고리 | 질문 예시 | 준비도 |
|---------|---------|-------|
| **A. 시스템 아키텍처** | ChatDev/MetaGPT 와 차이? | 시연 대본 Q1, Q2 참조 — 핵심 답변 머리에 |
| **B. RQ2 통신 프로토콜** | Protocol D 가중치 근거? | 시연 대본 Q4, Q5 — "초기값" 솔직히 |
| **C. RQ3 메모리** | ExpeL 과 차이? noisy 메모리 위험? | 시연 대본 Q6-Q8 — 한계 인정 |
| **D. 실험 / 검증** | 16GB 로 학술 의미? generalizability? | 시연 대본 Q9, Q10 — 자원 한계 정직 |
| **E. 시스템 / 구현** | DB 선택 / EventBus vs Queue? | 시연 대본 Q11, Q12 — 기술 선택 근거 |
| **F. Failure handling** | CTO 잘못된 분해 / 에러 전파? | 시연 대본 Q13 — Stage Gate + rework |

### 4.2 답변의 골격 (모든 질문에 적용)

```
1. 한 줄 요약 답 (5초)
   "네, [핵심 답]"

2. 근거 (15초)
   "그 이유는 [구체적 reasoning] 때문입니다"

3. 한계/단서 (10초)
   "다만 [현재 모르는 것 / 검증 필요한 것] 도 있어서
    이건 연구 중에 [구체적 검증 방법] 으로 확인할 계획입니다"
```

총 30초 정도가 이상적. 길어지면 청중 집중도 떨어짐.

### 4.3 정직성의 가치

랩실 발표에서 **모르는 걸 솔직히 인정하는 사람** 이 가장 신뢰받습니다. 학부생 레벨에서는 모르는 게 당연하고, 교수/선후배는 그걸 압니다. 화려한 말로 회피하면 즉시 들킵니다.

특히 다음 표현은 **신뢰도를 올립니다**:
- "솔직히 거기까지는 못 봤습니다"
- "추정으로는 X 인데 검증 안 한 부분입니다"
- "이건 대학원 와서 같이 풀어보고 싶은 부분입니다"
- "잘못 이해하고 있을 수도 있는데, 제가 파악한 한계는 [X] 입니다"

## 6. 강조 포인트 — 청중에게 남길 3가지 인상

### 인상 #1 — "시스템이 실제로 작동한다"
- 470+ 단위 테스트, 16GB Mac 동작 검증
- 시연 중 코드를 실제로 펼쳐서 보여줌
- 시스템 한계 (R1-Critical, B2-Critical 등) 도 사전에 파악되어 있음

> 시연 가능한 baseline 이 존재한다는 것이 RQ 의 신뢰도를 높입니다.

### 인상 #2 — "RQ 는 시스템에서 자연스럽게 도출됐다"
- 두 선행 시스템에서 식별된 두 한계 → RQ1 + RQ3
- 기존 논문 (ChatDev, ExpeL, ProtocolBench) 위치도 명확
- 임의 선택이 아니라 **시스템 운영 → 한계 식별 → 학술 질문** 의 자연스러운 도출 흐름

> 연구는 아이디어가 아니라 **문제** 에서 출발한다는 점을 시연을 통해 보여줍니다.

### 인상 #3 — "한계를 명확히 알고 있다"
- 16GB Mac 자원 한계 — 인지
- 검증 도메인 generalizability — 인지
- CEM noise 위험 — 인지
- Protocol D 가중치의 임의성 — 인지

> 인정한 한계는 보완 계획이 있는 한계입니다.

## 7. 발표 후 lab 적합성 검증을 위한 질문

발표가 끝나면 lab 의 연구 방향과 RQ 의 fit 을 검증하는 질문을 한두 개 준비:

> "현재 lab 에서 진행 중인 멀티에이전트 / 통신 관련 프로젝트가 있다면 RQ2 의 Protocol 비교 실험과 어떤 시너지가 있을지 듣고 싶습니다."

> "ExpeL 류 self-improving agent 연구와 RQ3 의 집단 메모리 방향성을 비교했을 때 lab 관점에서 더 흥미로운 axis 가 있을까요?"

이런 질문은 **연구 fit 검증** 자세를 보여주는 동시에, 청중의 답을 통해 RQ 우선순위를 조정할 단서를 얻을 수 있습니다.
