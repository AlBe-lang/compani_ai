## ❓ 예상 질문 + 답변 가이드

### A. 시스템 아키텍처 질문

#### Q1. ChatDev 과 무엇이 다른가요?
**답변**:
> "ChatDev 는 같은 LLM(GPT-4)을 역할만 바꿔서 반복 호출하는 구조입니다. 즉 Backend 에이전트도 Frontend 에이전트도 같은 GPT-4 인스턴스에 다른 system prompt 를 줘서 시뮬레이션합니다.
>
> 제 시스템은 **모델 자체를 다르게** 씁니다. CTO 는 qwen3:8b 같은 reasoning 모델, Backend/Frontend 는 gemma4:e4b 같은 코드 생성 특화 SLM, MLOps 는 llama3.2:3b 같은 가벼운 instruction-following 모델 — 역할에 맞는 모델을 매칭합니다. 모든 모델이 로컬 Ollama 에서 돌아서 비용이 0이고 16GB Mac 에서 재현 가능합니다.
>
> 그리고 ChatDev 는 통신을 자연어 토큰 그대로 주고받는데, 저는 SharedWorkspace + MessageQueue 의 구조화된 통신을 사용합니다. 그래서 통신 토큰이 ChatDev 보다 적게 들고, 무엇보다 — 그게 RQ2 의 출발점인데 — 어떤 통신 프로토콜이 정확성과 속도를 동시에 만족하는지 비교 실험할 수 있습니다."

#### Q2. MetaGPT 와는?
**답변**:
> "MetaGPT 는 SOP(Standard Operating Procedure) 기반으로 워크플로우가 사전 정의되어 있습니다. PRD → 시스템 디자인 → API 디자인 → 코드 순으로 고정된 단계를 따라갑니다.
>
> 제 시스템은 CTO 가 task decomposition 을 LLM 추론으로 동적으로 결정합니다. 즉 'Todo' 같이 단순한 요청은 5개 task, '블로그 플랫폼' 같이 복잡한 요청은 12개 task로 자동 늘어납니다. 그리고 의존성도 동적이라 dep_completed 시점에 다음 task 가 시작됩니다."

#### Q3. 왜 SLM 인가요? GPT-4 같은 LLM 한 개가 더 효율적이지 않나요?
**답변**:
> "이게 RQ1 의 핵심 질문이고, 실은 검증이 필요합니다. 제 가설은 **task 가 잘게 나뉘고 역할 전문성이 중요한 경우** SLM 팀이 같은 FLOPs 에서 LLM 단일보다 우위가 있다는 것입니다.
>
> 직관은 이렇습니다. GPT-4 한 번 호출은 비용이 크지만 매번 모든 컨텍스트를 처리합니다. 반면 SLM 5개를 병렬로 부르면 각자는 자기 task 컨텍스트만 처리하니 빠르고 가볍습니다. 단, 통신 오버헤드가 추가됩니다.
>
> 실험으로 확인하고 싶은 것은 어느 task 유형에서 break-even point 가 있는가입니다. ProtocolBench 2025 처럼 동일 FLOPs 조건으로 비교 실험을 설계하려고 합니다. 솔직히 결과는 모릅니다 — 가설 검증이 연구의 목적입니다."

### B. 통신 프로토콜 (RQ2) 질문

#### Q4. Protocol D 의 importance_score 가중치(0.4×관련성 + 0.3×최신성 + 0.3×심각도)는 어떻게 정했나요?
**답변**:
> "현재는 **임의의 초기값**입니다. 직관적으로 '관련성이 가장 중요하지만, 최근 정보와 심각한 정보도 무시할 수 없다' 라는 정도로만 잡은 hand-tuned 시작점입니다.
>
> 연구 과정에서 두 가지를 할 계획입니다:
> 1. **그리드 서치** — 가중치 조합을 0.1 단위로 변화시키며 task completion rate 와 통신 토큰 수의 trade-off 측정
> 2. **학습 기반** — 과거 task 결과 데이터셋이 쌓이면 가중치 자체를 메타러닝으로 최적화
>
> 다만 RQ2 의 본질은 가중치 자체보다 **'중요도 기반 선택적 통신' 이라는 패러다임이 자연어 전체 전송보다 효율적인가** 입니다. 가중치는 그 안의 implementation detail 입니다."

#### Q5. ProtocolBench 와 차별화 포인트는?
**답변**:
> "ProtocolBench 는 기존 프로토콜들을 비교하는 벤치마크입니다. 제가 거기 더하고 싶은 건 **(a) Protocol D 의 새로운 제안** 과 **(b) 집단 메모리와의 연계** 입니다.
>
> 특히 (b) 가 중요한데, 통신 프로토콜과 메모리는 사실 분리된 문제가 아니라고 봅니다. 어떤 정보를 어떻게 주고받느냐에 따라 메모리에 쌓이는 데이터의 품질도 달라집니다. 두 axis 를 같이 보는 연구는 아직 많지 않은 것 같습니다."

### C. Collective Error Memory (RQ3) 질문

#### Q6. ExpeL 과 무엇이 다른가요?
**답변**:
> "ExpeL 은 단일 에이전트의 경험 누적입니다. 한 에이전트가 task 들을 풀어가면서 자기 경험을 메모리에 쌓고 다음 task 에 활용합니다.
>
> 제가 하고 싶은 건 **팀 전체가 공유하는** 집단 메모리입니다. 예를 들어 backend 에이전트가 'Pydantic v2 에서 `from_orm` 대신 `model_validate` 써야 한다' 는 실수를 한 번 했다면, 다음에 frontend 에이전트가 type 정의 작업을 할 때도 그 정보를 사용 가능해야 합니다.
>
> 그리고 한 단계 더 나아가서, 같은 실수가 N번 이상 반복되면 **자동으로 방지 규칙으로 승격**해서 다음 에이전트의 system prompt 에 고정 삽입하는 방식을 실험하고 싶습니다. ExpeL 의 insight extraction 과 비슷하지만 팀 단위 + 자동 규칙화입니다."

#### Q7. 어떤 데이터를 메모리에 저장하나요? Hallucination 같은 것은 어떻게 식별?
**답변**:
> "현재 설계는 **(a) 명시적 실패** 와 **(b) 암묵적 실패** 두 가지를 구분합니다.
>
> (a) 명시적 실패는 에이전트가 던진 ErrorCode (`E-LLM-TIMEOUT`, `E-PARSE-JSON`, `E-DEPS-BLOCKED` 등) — 이건 자동 캡처됩니다. 코드/severity/context 함께 Qdrant 에 임베딩.
>
> (b) 암묵적 실패는 어렵습니다. peer review 에서 reviewer 가 지적한 코멘트를 활용할 계획입니다. severity MAJOR 이상 코멘트는 'author 가 이 부분에서 실수했다' 의 신호로 간주.
>
> 진짜 hallucination — 즉 reviewer 도 못 잡은 잘못된 정보 — 는 어렵습니다. 이건 사용자 피드백이나 외부 검증 (예: 생성된 코드의 실제 테스트 통과 여부) 으로 보완해야 합니다."

#### Q8. 메모리가 noisy 하게 쌓여서 오히려 false positive 를 일으키지는 않을까요?
**답변**:
> "정말 중요한 질문입니다. 현재는 두 가지로 방어를 생각합니다.
>
> 1. **임계값** — 최소 N번 (예: 3번) 이상 같은 패턴이 반복되어야 '방지 규칙'으로 승격. 한두 번 실수는 noise 로 간주.
> 2. **Confidence score** — 각 메모리에 confidence 를 매겨서, 검색 시 임계값 이상만 프롬프트에 주입.
>
> 다만 솔직히 말씀드리면 이 부분은 실제 데이터 없이는 튜닝이 어렵습니다. 실험 단계에서 noise injection 도 의도적으로 해서 robustness 를 확인할 계획입니다."

### D. 실험 / 검증 질문

#### Q9. 어떤 도메인에서 실험하나요? compani_ai 만으로는 generalizability 가 부족하지 않나요?
**답변**:
> "타당한 지적입니다. 두 도메인을 병렬로 진행할 계획입니다.
> - **compani_ai** — 코드 생성 (제가 만든 시스템이라 실험 통제 쉬움)
> - **OLaLA** — 가짜뉴스 판별 (작년 다른 task 형태로 같은 SLM 협업 검증)
>
> 다만 두 도메인만으로는 부족합니다. 가능하면 HumanEval 같은 표준 벤치마크나 ProtocolBench 의 task suite 도 추가로 검증하고 싶습니다. 다만 이건 실험 환경 (16GB Mac) 의 한계로 task scope 가 제한될 수 있어, 교수님 지도가 필요한 부분입니다."

#### Q10. 16GB Mac 으로 어떻게 학술적 의미 있는 실험이 가능한가요? GPU 클러스터 없이?
**답변**:
> "한계 인정합니다. 다만 두 가지 관점입니다.
>
> 1. **재현 가능성 강조** — 16GB Mac 환경 자체가 contribution 일 수 있습니다. 'High-end GPU 없이도 멀티에이전트 시스템이 실용적인가' 라는 질문은 실무자/오픈소스 커뮤니티에 의미가 있습니다.
>
> 2. **연구실 자원 활용 가능성** — 만약 연구실에서 더 큰 자원을 사용할 수 있다면 GPT-4 baseline 비교나 더 큰 모델 (Qwen-2.5 70B) 사용도 가능합니다. 16GB Mac 은 prototyping 환경으로 두고, 본 실험은 연구실 자원 활용을 희망합니다."

### E. 시스템 / 구현 질문

#### Q11. SharedWorkspace + MessageQueue 구조의 합리성은? Database 없이 메모리 dict 로도 가능하지 않나요?
**답변**:
> "두 가지 이유로 SQLite + Qdrant 조합을 선택했습니다.
>
> 1. **Persistence + Replay** — 파이프라인 중간 crash 가 나도 SQLite 에 저장된 WorkItem 상태로 재개 가능. 디버깅과 재현성에 중요합니다.
>
> 2. **벡터 검색** — Q3 의 집단 메모리에서 유사 task 검색이 필요한데 Qdrant 가 그 역할입니다. 그리고 Qdrant 의 collection 을 SharedWorkspace 의 query 로 노출하면 자연스럽게 통합됩니다.
>
> 메모리 dict 는 v0 prototype 일 때 사용했는데, async 환경에서 race condition 잡기 어렵고 persistence 가 없어서 디버깅이 매우 힘들었습니다. SQLite 로 옮기면서 EventBus + WAL 모드 조합으로 동시성 문제도 해결됐습니다."

#### Q12. 통신은 EventBus 인데 Q&A 는 MessageQueue. 두 채널을 분리한 이유는?
**답변**:
> "두 채널의 의미론이 다릅니다.
>
> - **EventBus (publish/subscribe)** — '상태 변화 알림'. 누가 듣든 상관없는 broadcast. 예: `task.completed` 는 모든 구독자에게 전파.
> - **MessageQueue (point-to-point)** — '특정 에이전트에게 응답을 요구하는 메시지'. Q&A 는 응답을 기다려야 하므로 큐의 dequeue 의미론이 맞습니다.
>
> 분리하지 않고 EventBus 로 통합할 수도 있지만, 그러면 'Q에 답이 왔는지' 추적을 application layer 에서 다 해야 합니다. SQLite-backed MessageQueue 가 그걸 자연스럽게 처리합니다."

#### Q13. Failure case 는 어떻게 처리하나요? CTO 가 잘못된 task 분해를 하면?
**답변**:
> "Stage Gate Meeting 이라는 메커니즘이 있습니다. 모든 task 가 끝난 후 (또는 task_count 의 절반이 끝난 후) CTO 가 모여서 'success rate 이 임계값 이상인가, avg duration 이 한도 이내인가' 를 평가합니다. 미달이면 abort 또는 rework.
>
> 다만 솔직히 말씀드리면 — CTO 가 처음부터 task 분해를 잘못한 경우는 아직 견고하지 않습니다. 시니어 QA 리뷰에서 식별된 R-09C (rework 시 dependency 재구성) 가 그 예입니다. 대학원 연구에서는 이 부분을 **runtime adaptation** 으로 강화하고 싶습니다 — task 실패 패턴을 메모리에 쌓아 다음 분해 시 CTO 가 참조."
