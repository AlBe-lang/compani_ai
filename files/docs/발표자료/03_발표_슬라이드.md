## 슬라이드 3 — 출발점 ① OLaLA (2024)

```
OLaLA — 가짜뉴스 판별 다중 에이전트 시스템

  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐
  │  Team A      │ →  │  Team B      │ →  │  Judge       │
  │  Stage 1–5   │    │  Stage 6–8   │    │  Stage 9     │
  │  증거 수집   │    │  이중 대립   │    │  최종 판정   │
  └──────────────┘    └──────────────┘    └──────────────┘
                          ↑↓
                    SLM 두 개가 같은 증거를
                    각각 지지/반박 관점으로 분석

  결과   • 단일 LLM 대비 hallucination ↓ 약 20%
         • 정확도 89% 달성
         • 역할 분리만으로 SLM 도 충분히 동작 → 첫 의문
```

**발표 멘트 (60초)**:
> "OLaLA 는 LangGraph 9-stage 파이프라인 기반 가짜뉴스 판별 시스템입니다. 핵심은 Stage 6-7 의 이중 대립 에이전트로, 같은 증거를 SLM 두 개가 한 쪽은 지지, 한 쪽은 반박하는 관점으로 독립 분석한 뒤 Judge 가 종합합니다. 결과적으로 단일 LLM 대비 hallucination 이 20% 줄고 정확도 89% 를 달성했습니다.
>
> 이 결과에서 한 가지 관찰점이 도출됩니다 — **SLM 도 역할만 잘 분리하면 협업이 가능한데, 왜 기존 멀티에이전트 연구 대부분은 단일 대형 모델 의존 구조인가?** 이게 본 연구의 첫 번째 motivation 입니다."

## 슬라이드 4 — 출발점 ② CompaniAI

```
CompaniAI — 사용자 한 문장 → 자동 소프트웨어 생성

  사용자: "Todo REST API 만들어줘"
                ↓
  CTO Agent     │  전략 수립 + 작업 분해 (자율적)
  (qwen3:8b)    │
                ↓
  Backend SLM   │  Frontend SLM   │  MLOps SLM
  (gemma4:e4b)  │  (gemma4:e4b)   │  (llama3.2:3b)
                ↓        ↕             ↓
       SharedWorkspace + MessageQueue
       (Q&A · Emergency Meeting · Peer Review)
                ↓
       outputs/ todo-rest-api/
       backend/ frontend/ deploy/    (~15 files)

  GitHub  AlBe-lang/compani_ai · 470+ 단위 테스트
          Mac Mini M4 16GB 동작 검증
```

**발표 멘트 (45초)**:
> "OLaLA 가 검증한 'SLM 협업의 가능성' 을 더 복잡한 도메인 — 코드 생성 — 으로 확장한 시스템이 CompaniAI 입니다. 사용자가 자연어 한 문장만 입력하면 CTO 가 전략과 작업 분해를 자율 결정하고, 역할별 SLM 들이 SharedWorkspace + MessageQueue 로 협업하며 실제 동작 가능한 코드를 생성합니다. 모든 모델이 로컬 Ollama 로 동작해 비용 0, 16GB Mac 환경에서 재현 가능합니다.
>
> 이 시스템을 운영하면서 또 다른 한계가 드러납니다."

## 슬라이드 5 — 두 가지 한계 (Research Motivation)

```
한계 1   단일 LLM 의존 구조

  ChatDev / MetaGPT 등 기존 멀티에이전트 연구도
  실제로는 GPT-4 한 모델을 system prompt 만 바꿔 반복 호출
  → 비용 ↑, 재현성 ↓, 단일 모델 한계 그대로 전이


한계 2   정적인 에이전트 — 경험 누적 부재

  실행 중 hallucination / schema violation 이 발생해도
  같은 패턴이 다음 실행에서도 반복
  → ExpeL 등이 단일 에이전트 경험 누적은 다뤘으나
    팀 단위 집단 메모리는 미연구 영역

────────────────────────────────────────────────
이 두 한계의 교차점이 본 연구의 출발점입니다.
```

**발표 멘트 (45초)**:
> "두 한계는 학계가 충분히 다루지 않은 영역입니다. 첫 번째는 ChatDev, MetaGPT 등이 형식상 멀티에이전트지만 단일 LLM 의 system prompt 만 바꿔 반복 호출하는 구조라 비용과 재현성에 한계가 있다는 점.
>
> 두 번째는 ExpeL 이 단일 에이전트 경험 누적은 다뤘지만 — 팀 전체가 공유하는 episode 단위 메모리는 미연구 영역이라는 점입니다. 같은 종류의 hallucination 이 task 마다 반복되는 현상이 실제로 관측됩니다.
>
> 이 두 한계의 교차점이 RQ 의 출발점입니다."

## 슬라이드 8 — 시스템 아키텍처 (4 계층)

```
┌───────────────────────────────────────────────────────────┐
│ L1  오케스트레이션  · CTO (qwen3:8b)                      │
│     전략 → 분해 → 모니터링 → 메모리 조회                  │
├───────────────────────────────────────────────────────────┤
│ L2  실행  · 역할별 SLM 병렬                               │
│     Backend (gemma4:e4b)                                  │
│     Frontend (gemma4:e4b)    asyncio + concurrency limiter│
│     MLOps (llama3.2:3b)                                   │
├───────────────────────────────────────────────────────────┤
│ L3  통신  ★ RQ2                                           │
│     SharedWorkspace · MessageQueue ·                      │
│     Emergency Meeting · Peer Review                       │
├───────────────────────────────────────────────────────────┤
│ L4  집단 실수 메모리 (CEM)  ★ RQ3  [미구현 — 석사 1순위] │
│     Qdrant (벡터) + SQLite (메타) + 자동 규칙 승격        │
└───────────────────────────────────────────────────────────┘
```

**발표 멘트 (60초)**:
> "현재 구현은 L1, L2, L3 까지 작동합니다. 약 470 개의 단위 테스트가 통과한 상태이고 GitHub 에 공개되어 있습니다.
>
> 이 위에 두 가지 확장이 본 연구의 핵심입니다. 하나는 L3 의 통신 계층을 4 가지 프로토콜로 추상화해서 비교 실험할 수 있게 만드는 것 — RQ2. 다른 하나는 L4 집단 실수 메모리를 처음부터 새로 설계하는 것 — RQ3. 그리고 두 axis 의 위에 RQ1 — SLM 팀이 단일 LLM 대비 어떤 task 에서 우위인가 — 가 메타 질문으로 자리합니다."

## 슬라이드 9 — RQ1 (모델 차원)

```
RQ1   동일 FLOPs 조건에서, 역할 특화 다중 SLM 팀은
      단일 LLM 대비 어떤 task 유형에서 우위인가?

가설  Task 가 (a) 명확히 분해 가능하고
      (b) 역할 전문성이 중요할수록 → SLM 팀 우위

비교  Llama 3.1 70B (단일)  vs  qwen3:8b + gemma4:e4b×2 + llama3.2:3b
                                         (~17B 합산)

측정  • 정확도 / 완성도
      • End-to-end latency
      • FLOPs / 토큰 비용
      • Cross-task 일반화
```

## 슬라이드 10 — RQ2 (통신 차원) ★

```
RQ2   에이전트 간 통신에서 정확성과 속도를 동시에
      만족하는 프로토콜의 설계 원칙은?

  Protocol A   자연어 전송         자연스럽지만 토큰 비효율
  Protocol B   JSON 구조화         현재 compani_ai 사용
  Protocol C   CoT 요약 공유       투명, 토큰 증가
  Protocol D ★ Importance-filtered 신규 제안

  Protocol D 의 점수 (초기값):
    importance(msg) = 0.4·relevance + 0.3·recency + 0.3·severity

  → 그리드 서치 + 메타러닝으로 가중치 최적화
```

**발표 멘트 (60초)**:
> "Protocol D 가 본 연구의 신규 제안입니다. 모든 메시지를 다 broadcast 하는 대신, 각 메시지에 중요도 점수를 매겨서 임계값 이상만 다음 에이전트에게 전달합니다. 점수는 관련성, 최신성, 심각도 세 요소의 가중합으로 시작하고, 가중치는 그리드 서치로 최적화할 계획입니다. 단순한 가중치 튜닝이 아니라 — 더 본질적으로 — '중요도 기반 선택적 통신' 이라는 패러다임이 자연어 전체 전송보다 효율적인지가 RQ2 의 질문입니다."

## 슬라이드 11 — RQ3 (메모리 차원) ★

```
RQ3   팀 전체가 공유하는 episode 단위 실수 메모리는
      반복 오류율을 얼마나 감소시키는가?

데이터  ErrorEvent { error_code, agent_id, task_id,
                    context_embedding, severity,
                    recurrence_count }

흐름   ┌──────────┐  유사도   ┌──────────┐
       │ 실수 발생│ ────────→│ Qdrant   │
       │ (자동)   │           │ + SQLite │
       └──────────┘           └────┬─────┘
                                   │ N≥3 반복
                                   ↓
                          🔔 자동 방지 규칙 생성
                          (모든 SLM 의 system prompt 에 고정 삽입)

비교   none / per-agent / shared CEM / shared CEM + auto-rule
       → 반복 오류율 감소 곡선 측정
```

**발표 멘트 (60초)**:
> "ExpeL 이 단일 에이전트의 경험 누적을 다뤘다면, 본 연구는 팀 전체가 공유하는 **집단** 메모리입니다. 한 에이전트가 한 번 실수한 패턴은 다른 에이전트가 다음 task 에 시작할 때 자동으로 시스템 프롬프트에 주입됩니다. 그리고 같은 패턴이 N=3 번 이상 반복되면 자동으로 '방지 규칙'으로 승격해서 모든 에이전트의 시스템 프롬프트에 고정됩니다. 이 2-tier 구조 — episodic + rule — 이 RQ3 의 핵심 설계입니다."

## 슬라이드 15 — 실험 3 매트릭스 (RQ3)

```
실험 3 — 집단 실수 메모리 효과 (8-10개월차)

╔═══════════════════════╤══════════════════════════════╗
║  메모리 조건           │  반복 오류율 (10회 실행 후)  ║
╠═══════════════════════╪══════════════════════════════╣
║  (a) 없음             │   기준선                      ║
║  (b) per-agent        │   감소 정도?                  ║
║  (c) shared CEM       │   감소 정도?                  ║
║  (d) shared + 자동규칙│   추가 감소?                  ║
╚═══════════════════════╧══════════════════════════════╝

가설
  H3-1   (a) → (c) 단조 감소
  H3-2   (c) vs (d) 에서 (d) 가 추가 감소
  H3-3   Cross-agent 오류는 (c) 가 (b) 대비 큰 감소

False positive  방지 규칙이 잘못 학습된 비율 — 안전장치 검증
```

## 슬라이드 20 — 백업: 시스템 상세 (질문 시)

```
SharedWorkspace 의 의미론

  - 모든 에이전트는 서로 직접 호출 안 함
  - 오직 SharedWorkspace 만 read/write
  - WorkItem 상태 변화 → EventBus 로 broadcast
  - 의존성 있는 에이전트는 WorkItem.completed 이벤트 까지 polling

MessageQueue Q&A 의미론

  - point-to-point 메시지
  - SQLite-backed, Q 발생 시 응답 받기까지 dequeue 보장
  - asyncio.wait_for 로 timeout 보호

Emergency Meeting 트리거

  - blocking.detected (의존성 충돌)
  - peer_review.severity ≥ MAJOR
  - stage_gate.failure
```
